问题：详细给我讲讲Redis、队列、Nginx

这三个在后台系统里非常常见，尤其是你现在接触的 **后台管理系统、设备列表、学校库、批量导入、报表生成、接口性能优化、域名部署**，都会遇到。

你可以先用一句话理解：

```text id="4mn9ku"
Redis：高速临时仓库，让系统更快。
队列：任务排队系统，让耗时任务慢慢处理。
Nginx：系统入口门卫，把请求转发到正确服务。
```

它们不是同一类东西，作用完全不同。

---

# 一、先放到完整系统链路里理解

假设用户访问你的系统：

```text id="n584fq"
用户浏览器 / APP / 小程序
↓
域名
↓
Nginx
↓
前端页面 / 后端接口
↓
后端服务
↓
Redis / MySQL / 队列
↓
Worker 后台任务
```

可以理解成：

| 组件    | 位置            | 核心作用                  |
| ----- | ------------- | --------------------- |
| Nginx | 系统入口          | 接收请求、转发请求、处理 HTTPS、分流 |
| Redis | 后端旁边          | 存缓存、临时数据、登录态、锁、热点数据   |
| 队列    | 后端和 Worker 中间 | 把耗时任务排队，异步处理          |

再直白一点：

```text id="9f0n1c"
Nginx：用户请求先进来的大门
Redis：后端旁边的高速临时仓库
队列：后台任务的排队机
```

---

# 二、Redis 是什么？

## 1. Redis 的本质

Redis 是一个**内存型数据存储系统**。

你可以把它理解成：

> **Redis 是一个速度很快的临时数据仓库。**

它和 MySQL 最大的区别是：

| 对比     | MySQL       | Redis            |
| ------ | ----------- | ---------------- |
| 主要存储位置 | 硬盘          | 内存               |
| 速度     | 相对慢         | 非常快              |
| 数据结构   | 表、行、字段      | key-value、列表、集合等 |
| 适合存什么  | 核心业务数据      | 缓存、临时数据、热点数据     |
| 是否长期保存 | 是           | 不一定              |
| 典型用途   | 用户、订单、设备、学校 | 验证码、缓存、登录态、排行榜   |

所以不要把 Redis 简单理解成数据库替代品。

更准确地说：

> **MySQL 存正式业务数据，Redis 存高频访问或临时状态数据。**

---

# 三、Redis 解决什么问题？

Redis 主要解决 5 类问题。

---

## 1. 缓存：减少数据库压力

比如你的学校库有一个“国家地区列表”。

国家地区数据变化很少，但用户每次打开筛选器都要查。

如果每次都查 MySQL：

```text id="81mv8o"
用户打开页面
↓
后端查 MySQL 国家列表
↓
返回前端
```

访问量大了，数据库压力会增加。

如果用 Redis 缓存：

```text id="5jgpmd"
第一次请求：
后端查 MySQL
↓
把结果存到 Redis
↓
返回前端

后续请求：
后端直接查 Redis
↓
返回前端
```

Redis 很快，所以接口响应会更快。

适合缓存的数据：

```text id="pvt8mr"
国家地区列表
设备类型枚举
建设状态枚举
学校筛选项
首页配置
会员权益配置
热门文章
用户菜单权限
系统字典
```

不太适合长期缓存的数据：

```text id="2u6epr"
订单支付状态
账户余额
设备实时状态
库存
强实时财务数据
```

因为这些数据一旦延迟，可能影响业务正确性。

---

## 2. 验证码、登录态、Token

短信验证码通常只需要保存 5 分钟。

比如：

```text id="xay2oa"
手机号：138xxxx
验证码：123456
有效期：5分钟
```

这种数据不适合长期存 MySQL，适合放 Redis。

Redis 可以设置过期时间：

```text id="4h99v9"
code:138xxxx = 123456
TTL = 300秒
```

5 分钟后自动过期。

适合 Redis 的临时数据：

```text id="0da0yn"
短信验证码
图形验证码
登录 token
临时授权码
找回密码验证码
小程序 session
```

---

## 3. 防重复提交

比如用户点击“生成报表”。

如果用户连续点 5 次，系统可能生成 5 个一模一样的报表任务。

可以用 Redis 做一个短期标记：

```text id="whw1nz"
用户A + 月报 + 2026-06 = 正在生成中
```

在 Redis 里设置：

```text id="1hpz0g"
report:userA:2026-06 = processing
过期时间：10分钟
```

如果用户重复点击，系统提示：

```text id="hjw4pf"
该报表正在生成中，请稍后查看
```

这就是用 Redis 防重复提交。

常见场景：

```text id="j8x23a"
重复支付
重复提交审核
重复生成报表
重复导入文件
重复发送验证码
重复点击按钮
```

---

## 4. 分布式锁

这个稍微进阶，但产品经理要理解。

假设有多个后端服务同时处理同一个任务。

比如：

```text id="f0c6yr"
Worker A 正在处理导入任务 1001
Worker B 也拿到了导入任务 1001
```

如果两个 Worker 同时执行，可能导致重复导入。

这时可以用 Redis 加锁：

```text id="zsfqeq"
任务1001正在被处理，其他 Worker 不允许处理
```

这叫分布式锁。

适合场景：

```text id="e07bao"
防止同一订单重复支付处理
防止同一报表重复生成
防止同一批设备重复导入
防止同一任务被多个 Worker 同时执行
```

产品经理不用知道具体代码，但要知道：

> 涉及重复执行、并发处理、金额权益、导入任务时，要考虑“是否需要防重复处理”。

---

## 5. 限流

限流就是限制某个用户或接口访问频率。

比如短信验证码：

```text id="5dut8e"
同一手机号 60 秒内只能发送 1 次
同一手机号 1 天最多发送 10 次
```

可以用 Redis 记录次数：

```text id="9wll3c"
sms:138xxxx:count = 3
```

适合限流的场景：

```text id="cct2mm"
短信验证码
登录失败次数
AI问答调用次数
导出报表次数
第三方接口调用次数
设备控制指令频率
```

比如 AI 问答每次调用都有成本，产品上可以设计：

```text id="kdgdqe"
普通用户每天最多提问 20 次
会员用户每天最多提问 200 次
```

Redis 就很适合做这种计数。

---

# 四、Redis 的典型使用场景

结合你的业务，Redis 可以用在这些地方。

## 学校库 / 内容平台

```text id="4w1ten"
国家列表缓存
地区列表缓存
课程类型缓存
文章分类缓存
热门文章缓存
会员权限缓存
用户登录态
AI问答次数限制
```

比如国家地区数据不常变，可以缓存 24 小时。

---

## HEMS / 设备系统

```text id="4k72ni"
设备最新在线状态缓存
设备最后上报时间缓存
首页能源概览缓存
设备告警数量缓存
用户电站权限缓存
设备控制指令防重复
```

注意：设备实时数据是否放 Redis，要看实时性要求。

比如“设备最后上报时间”可以放 Redis 加速判断，但最终历史数据仍然应该入库。

---

## 后台管理系统

```text id="6tu54m"
菜单权限缓存
数据字典缓存
导入任务防重复
报表生成状态缓存
登录 token
验证码
```

# 五、Redis 的风险和产品注意点

Redis 不是用了就一定好，它有几个典型问题。

---

## 1. 缓存不是最新

比如后台修改了学校分类，但 Redis 里还缓存着旧分类。

用户看到的可能还是旧数据。

所以产品经理要问：

```text id="gluwga"
缓存多久过期？
后台修改后是否立即刷新缓存？
是否需要手动刷新缓存？
用户能不能接受数据延迟？
```

PRD 可以写：

```text id="1e1wto"
国家地区筛选项允许缓存，缓存时间 24 小时。
后台修改国家地区后，需主动刷新缓存，确保前台下一次访问展示最新数据。
```

---

## 2. 缓存穿透

用户一直查询不存在的数据。

比如请求：

```text id="7uw4bj"
school_id = -999999
```

数据库查不到，Redis 也查不到。

如果大量恶意请求过来，会一直打到数据库。

解决思路：

```text id="4d78sg"
不存在的数据也短时间缓存为空结果
```

产品经理不需要深入，但要知道：

> 高频查询、搜索、详情接口，需要考虑异常请求和恶意请求。

---

## 3. 缓存雪崩

大量缓存同一时间过期，所有请求突然打到数据库。

比如 10000 个缓存都设置 00:00 过期。

00:00 一到，数据库压力暴增。

解决思路：

```text id="buwd0i"
缓存过期时间错开
热点数据不过期或主动刷新
```

---

## 4. Redis 宕机怎么办？

如果 Redis 挂了，可能导致：

```text id="59z46h"
验证码不可用
登录态异常
缓存失效
接口变慢
队列异常
```

产品上要关注：

```text id="zi33hv"
Redis异常时，核心业务是否还能用？
是否降级到数据库查询？
是否提示系统繁忙？
是否影响支付、订单、设备控制？
```

核心原则：

> Redis 不能成为关键业务唯一数据源，核心数据仍应落 MySQL 或其他正式存储。

---

# 六、队列是什么？

## 1. 队列的本质

队列就是：

> **任务排队系统。**

你可以把它理解成餐厅排号。

用户来了，不是所有人同时进厨房，而是先排队：

```text id="mnzixy"
任务1：批量导入学校
任务2：生成月报
任务3：AI翻译文章
任务4：发送短信
```

后台 Worker 像厨师：

```text id="9bl44f"
Worker 取出任务1处理
处理完再处理任务2
```

如果 Worker 多，就可以同时处理多个任务。

---

# 七、为什么需要队列？

队列主要解决 6 个问题。

---

## 1. 异步处理耗时任务

比如批量导入 10 万条设备。

同步方式：

```text id="gx318x"
用户上传文件
↓
接口一直处理
↓
用户一直等
↓
可能超时
```

队列方式：

```text id="jrq91u"
用户上传文件
↓
系统创建任务
↓
任务进入队列
↓
接口立即返回“任务已提交”
↓
Worker 后台慢慢处理
```

适合队列的任务：

```text id="z3p0a0"
批量导入
生成报表
AI翻译
发送短信
发送邮件
图片压缩
学校抓取
设备数据清洗
大文件处理
```

---

## 2. 削峰填谷

假设 100 个用户同时生成报表。

如果没有队列：

```text id="voh88s"
100 个报表同时查数据库
↓
数据库压力暴增
↓
系统变慢甚至崩溃
```

有队列：

```text id="rixxs3"
100 个报表进入队列
↓
系统每次只处理 5 个
↓
其他任务排队等待
```

这叫削峰填谷。

产品经理要理解：

> 队列不是让任务立即完成，而是让系统稳定、有节奏地处理任务。

---

## 3. 失败重试

比如 AI 翻译接口失败，可能是网络波动。

队列可以设置：

```text id="6ebqgn"
失败后 1 分钟重试
再失败 5 分钟重试
最多重试 3 次
```

适合重试的失败：

```text id="tf3v0l"
网络超时
第三方接口临时不可用
AI接口限流
文件上传失败
短信接口临时异常
```

不适合重试的失败：

```text id="gtlcil"
Excel格式错误
手机号格式错误
设备SN重复
用户无权限
必填字段为空
```

产品经理要区分：

```text id="wummb5"
系统异常可以重试
业务错误不应该盲目重试
```

---

## 4. 解耦系统

解耦就是让两个系统不用强绑定。

比如用户注册成功后，需要做很多事：

```text id="eljqo7"
创建用户
发欢迎短信
发邮件
初始化会员权益
同步CRM
写操作日志
```

如果全部在注册接口里做，注册会很慢。

更好的方式：

```text id="kgur4b"
注册接口只负责创建用户
↓
创建成功后发送一条“用户注册成功”的消息
↓
短信服务、邮件服务、CRM同步服务各自处理
```

这样用户注册体验更快，系统也更稳定。

---

## 5. 控制并发

有些任务不能同时跑太多。

比如：

```text id="ehzqa1"
同时生成 100 个大报表，数据库扛不住
同时抓取 1000 个学校，目标网站可能封 IP
同时调用 AI 翻译，费用和限流都受影响
```

队列可以控制：

```text id="i92gpq"
最多同时处理 5 个报表任务
最多同时抓取 2 个学校任务
最多同时调用 10 个 AI 翻译任务
```

---

## 6. 保留任务过程和结果

队列任务通常会配合任务表。

可以记录：

```text id="5z65yc"
任务状态
任务进度
失败原因
重试次数
创建人
开始时间
完成时间
结果文件
```

这对于批量导入、报表生成非常重要。

---

# 八、队列的核心组成

一个完整队列系统通常包括：

```text id="5dzoq3"
生产者 Producer
队列 Queue
消费者 Consumer / Worker
任务数据 Job
任务状态 Task Status
失败重试 Retry
死信队列 Dead Letter Queue
```

你作为产品经理主要理解这几个就够。

---

## 1. Producer：生产者

生产者就是创建任务的一方。

例如：

```text id="yyslfb"
用户点击“生成报表”
↓
后端创建一个报表任务
↓
把任务放进队列
```

后端就是生产者。

---

## 2. Queue：队列

队列就是任务等待区。

```text id="3lsmim"
任务1 等待处理
任务2 等待处理
任务3 等待处理
```

---

## 3. Worker：消费者

Worker 是真正干活的后台程序。

```text id="ntgyus"
Worker 取任务
↓
查数据库
↓
生成文件
↓
上传文件
↓
更新任务状态
```

---

## 4. Retry：重试

任务失败后，根据规则重新执行。

产品经理要定义：

```text id="qwc13d"
失败后是否重试？
最多重试几次？
间隔多久？
哪些失败不重试？
失败后用户看到什么？
```

---

## 5. Dead Letter Queue：死信队列

如果任务重试多次仍失败，就进入死信队列。

可以理解成：

> 处理不了的问题任务，放到异常任务池，等待人工排查。

适合：

```text id="hsp3vi"
多次发送失败的短信
多次生成失败的报表
多次同步失败的第三方数据
```

---

# 九、队列的业务场景举例

## 1. 批量导入设备

```text id="iiaxs4"
用户上传 Excel
↓
后端保存文件
↓
创建导入任务
↓
任务进入队列
↓
Worker 解析 Excel
↓
校验每行数据
↓
分批写入数据库
↓
生成失败明细
↓
更新任务状态
```

产品要定义：

```text id="1m2wj1"
是否允许部分成功？
失败明细包含哪些字段？
是否支持重新导入失败数据？
重复设备怎么处理？
导入中是否允许再次上传？
```

---

## 2. 生成报表

```text id="sj5jb3"
用户选择报表条件
↓
点击生成
↓
系统创建报表任务
↓
任务进入队列
↓
Worker 查询数据
↓
计算统计
↓
生成 Excel
↓
上传 COS
↓
返回下载链接
```

产品要定义：

```text id="d637l2"
报表文件保存多久？
相同条件是否重复生成？
生成失败是否可重试？
普通用户能否看别人的报表？
管理员能否查看全部任务？
```

---

## 3. 学校抓取和 AI 翻译

你的学校库系统非常适合用队列。

```text id="mfwst9"
选择学校
↓
创建抓取任务
↓
队列排队
↓
Worker 抓取学校详情
↓
下载图片
↓
AI翻译
↓
生成待审核版本
```

为什么适合队列？

```text id="mhqmd5"
抓取耗时不可控
目标网站可能限流
AI翻译有成本和限频
图片下载可能失败
任务需要失败重试
```

---

## 4. 设备数据清洗

IoT 系统中，设备会上报大量数据。

如果每条上报都同步做复杂计算，系统压力很大。

可以：

```text id="6ved3f"
设备上报原始数据
↓
先快速接收并存储
↓
把数据清洗任务放进队列
↓
Worker 后台计算统计、告警、能流数据
```

---

# 十、队列的产品风险

队列也不是万能的。

## 1. 队列积压

如果任务进来太多，Worker 处理不过来，就会积压。

表现：

```text id="qvo9xj"
任务一直排队中
报表很久不完成
导入任务长时间处理中
短信延迟发送
```

产品侧要考虑提示：

```text id="0vrvog"
当前任务较多，预计需要较长时间，请稍后查看
```

也要问开发：

```text id="yjuesd"
队列积压时是否有告警？
是否可以增加 Worker？
是否有任务优先级？
```

---

## 2. 任务重复执行

队列任务可能因为重试、超时、Worker 重启而重复执行。

所以任务要考虑幂等。

比如设备导入，不能重复插入相同 SN。

产品要定义：

```text id="iwnexq"
重复数据是跳过、覆盖、更新，还是报错？
```

---

## 3. 任务失败用户不知道

异步任务不能让用户提交后就消失。

必须有任务中心或结果通知。

产品上至少要有：

```text id="elyp1x"
任务列表
任务状态
失败原因
结果下载
重试入口
```

---

## 4. 任务顺序问题

有些任务有先后顺序。

比如：

```text id="l18oat"
先导入学校基础信息
再导入学校图片
再导入课程信息
```

如果顺序错了，可能关联失败。

产品和开发要确认：

```text id="wrwkmb"
任务之间是否有依赖关系？
是否必须按顺序处理？
某一步失败后后续任务是否继续？
```

---

# 十一、Nginx 是什么？

## 1. Nginx 的本质

Nginx 是一个高性能 Web 服务器和反向代理服务器。

你可以把它理解成：

> **Nginx 是系统入口的门卫和分流器。**

用户请求进来后，Nginx 决定：

```text id="3vwj3e"
这个请求访问哪个项目？
这个请求转给哪个后端服务？
这个域名是否允许访问？
这个请求是 HTTP 还是 HTTPS？
这个路径应该返回前端页面还是接口数据？
```

---

# 十二、Nginx 解决什么问题？

Nginx 主要解决 7 类问题。

---

## 1. 静态资源服务

前端项目打包后通常是一堆静态文件：

```text id="9ad1g7"
index.html
app.js
style.css
图片
字体
```

Nginx 可以直接把这些文件返回给浏览器。

比如用户访问：

```text id="yadfjd"
https://gloway.top
```

Nginx 返回前端页面。

---

## 2. 反向代理

反向代理是 Nginx 最常见作用。

用户请求：

```text id="2rqkji"
https://gloway.top/api/user/list
```

Nginx 转发给后端：

```text id="d457lv"
http://127.0.0.1:3000/api/user/list
```

用户不需要知道后端真实端口。

你之前的场景就是这样：

```text id="dmxrog"
/api → 主系统后端
/school-intelligence-api → 学校情报系统后端
```

这就是 Nginx 的路径转发。

---

## 3. 多项目共用一台服务器

一台服务器上可能跑多个服务：

```text id="ft1fon"
主系统前端
主系统后端 3000
测试系统后端 3001
学校情报系统后端 3010
学校情报测试后端 3011
```

Nginx 可以根据域名和路径分流：

```text id="oe4s1l"
gloway.top/api → 3000
test.gloway.top/api → 3001
gloway.top/school-intelligence-api → 3010
test.gloway.top/school-intelligence-api → 3011
```

没有 Nginx，用户可能需要访问：

```text id="gy4rrn"
http://服务器IP:3000
http://服务器IP:3010
```

这不适合正式产品。

---

## 4. HTTPS 证书

用户访问：

```text id="4rj9a7"
https://gloway.top
```

Nginx 通常负责处理 SSL 证书。

它负责：

```text id="7webj9"
加载证书文件
支持 HTTPS
HTTP 跳转 HTTPS
校验证书域名
```

如果证书配置错，可能出现：

```text id="gw6tuu"
HTTPS访问失败
证书不可信
证书和域名不匹配
443端口不可用
```

---

## 5. HTTP 跳转 HTTPS

产品上通常希望用户都用 HTTPS。

Nginx 可以配置：

```text id="a1uqur"
http://gloway.top
↓
https://gloway.top
```

这叫 HTTP 强制跳 HTTPS。

---

## 6. 负载均衡

如果一个后端服务扛不住，可以部署多个实例。

比如：

```text id="ooal5t"
后端服务A：3000
后端服务B：3002
后端服务C：3003
```

Nginx 可以把请求分给不同服务：

```text id="9ld0l6"
第1个请求 → 服务A
第2个请求 → 服务B
第3个请求 → 服务C
```

这叫负载均衡。

产品经理不一定要深入，但要知道：

> 当系统访问量大时，Nginx 可以帮忙把流量分摊到多个后端服务。

---

## 7. 限流和安全控制

Nginx 也可以做一些入口层限制：

```text id="gnhlis"
限制请求频率
限制上传文件大小
屏蔽某些 IP
禁止访问敏感路径
配置跨域
设置请求超时时间
```

比如上传文件时，如果 Nginx 限制了最大 10MB，而用户上传 20MB，就会失败。

产品经理遇到上传失败时，要想到：

```text id="u9wlbb"
前端限制了吗？
后端限制了吗？
Nginx限制了吗？
文件存储限制了吗？
```

---

# 十三、Nginx 和域名、路由的关系

你之前问过域名和路由，这里正好串起来。

访问：

```text id="wbv0kc"
https://test.gloway.top/api/user/list
```

完整链路是：

```text id="0k96ec"
1. 浏览器访问 test.gloway.top
2. DNS 找到服务器 IP
3. 请求到达服务器
4. Nginx 接收请求
5. Nginx 根据域名 test.gloway.top 判断这是测试环境
6. Nginx 根据路径 /api 判断要转发给测试后端
7. 后端处理接口
8. 返回数据
```

所以：

```text id="24o7zx"
DNS：负责域名指向哪台服务器
Nginx：负责服务器内部转发到哪个项目
后端路由：负责接口内部由哪个代码处理
前端路由：负责页面跳转
```

这几个“路由”容易混淆。

---

# 十四、Nginx 常见问题

## 1. 502 Bad Gateway

常见原因：

```text id="xe9tdf"
后端服务没启动
Nginx代理的端口错了
后端服务崩了
Nginx连不上后端
```

产品经理可以理解为：

> Nginx 找到了入口，但后端服务没正常响应。

比如：

```text id="zwwdgy"
Nginx 转发到 127.0.0.1:3001
但 3001 服务没启动
```

就容易 502。

---

## 2. 504 Gateway Timeout

常见原因：

```text id="6x5w0r"
后端处理太慢
数据库查询太慢
第三方接口太慢
Nginx等待超时
```

产品经理可以理解为：

> Nginx 等后端等太久，放弃了。

这时要查：

```text id="3sjfes"
接口是否超时？
SQL是否慢？
是否应该改成异步任务？
Nginx超时时间是否太短？
```

---

## 3. 404 Not Found

常见原因：

```text id="pbc4y5"
路径不存在
前端路由没配置
Nginx路径匹配错
后端接口路径错
```

比如：

```text id="vvqbai"
/school-intelligence-api
```

写成：

```text id="rt8wcw"
/school-intelligence
```

就可能 404。

---

## 4. HTTPS 访问失败

常见原因：

```text id="efx1iu"
证书文件路径错
证书过期
证书不包含当前域名
443端口没开放
Nginx SSL配置错
```

---

## 5. 上传文件失败

可能原因：

```text id="drolra"
Nginx限制文件大小
后端限制文件大小
前端限制文件大小
COS/OSS限制文件大小
网络中断
```

Nginx 常见会限制请求体大小。

# 十五、三个组件放在一起举例

## 场景：用户生成电站月报

完整流程：

```text id="6v6jgz"
用户访问报表页面
↓
请求进入 Nginx
↓
Nginx 把 /api/report/create 转发给后端
↓
后端校验用户权限
↓
后端用 Redis 判断是否重复提交
↓
后端创建报表任务
↓
任务进入队列
↓
接口返回：报表生成中
↓
Worker 从队列取任务
↓
Worker 查询数据库并生成 Excel
↓
文件上传 COS
↓
任务状态更新为成功
↓
用户下载报表
```

这里三者作用分别是：

```text id="8hwgy7"
Nginx：把用户请求转发到后端
Redis：防重复提交、缓存权限、记录临时状态
队列：让报表生成任务后台排队处理
```

---

## 场景：用户批量导入设备

```text id="9bl8dt"
用户上传 Excel
↓
Nginx 接收上传请求，并转发给后端
↓
后端保存文件
↓
Redis 判断是否已有相同文件正在导入
↓
后端创建导入任务
↓
任务进入队列
↓
Worker 解析 Excel
↓
分批写入 MySQL
↓
失败明细上传 COS
↓
用户查看导入结果
```

这里要注意：

```text id="gwwoq0"
Nginx 可能限制上传大小
Redis 可以防重复导入
队列负责异步处理导入
MySQL 存最终设备数据
COS 存上传文件和失败明细
```

---

## 场景：学校库首页加载

```text id="3oeqe5"
用户打开小程序首页
↓
Nginx 转发 API 请求
↓
后端查询首页配置
↓
先查 Redis 是否有缓存
↓
Redis 有数据，直接返回
↓
Redis 没数据，再查 MySQL
↓
查到后写入 Redis
↓
返回前端
```

这里 Redis 的作用是提升访问速度。

# 十六、三者区别总结

| 对比        | Redis           | 队列            | Nginx               |
| --------- | --------------- | ------------- | ------------------- |
| 本质        | 内存型数据存储         | 任务排队系统        | 请求入口和反向代理           |
| 主要解决      | 快速读取、临时状态       | 耗时任务、异步处理     | 域名访问、路径转发、HTTPS     |
| 工作位置      | 后端旁边            | 后端和 Worker 中间 | 系统入口                |
| 典型场景      | 缓存、验证码、限流、锁     | 导入、报表、短信、AI翻译 | /api 转发、HTTPS、负载均衡  |
| 是否存正式业务数据 | 通常不存            | 不存，存任务消息      | 不存业务数据              |
| 是否用户直接感知  | 间接感知为更快         | 感知为任务处理中      | 感知为能不能访问            |
| 出问题表现     | 接口变慢、验证码异常、登录异常 | 任务卡住、处理中很久    | 502、504、404、HTTPS失败 |

```text id="vqwo6v"
是否允许部分成功？
失败是否重试？
是否有任务积压？
是否需要任务通知？
```

## Nginx 你要关注

```text id="yq576i"
域名是否正确？
HTTP是否跳HTTPS？
接口路径是否转发正确？
测试环境和生产环境是否隔离？

证书是否覆盖当前域名？
```

# 十九、PRD 里可以怎么体现

## Redis 相关写法

```text id="o3cni4"
国家地区、设备类型、建设状态等低频变更数据允许缓存，缓存时间为 24 小时。
后台修改相关配置后，应主动刷新缓存。
若缓存读取失败，系统降级为数据库查询，不影响用户正常访问。
```

---

## 队列相关写法

```text id="a9abxn"
批量导入采用异步任务处理。
用户上传文件后，系统立即创建导入任务并返回“任务已提交”。
任务状态包括：排队中、处理中、导入成功、部分成功、导入失败。
导入完成后展示成功数、失败数，并支持下载失败明细。
```

---

## Nginx 相关写法

```text id="q3plmk"
生产环境访问域名为 https://gloway.top。
HTTP 访问需自动跳转至 HTTPS。
/api 请求转发至主业务后端服务。
/school-intelligence-api 请求转发至学校情报系统后端服务。
测试环境与生产环境接口需隔离，避免测试数据写入生产库。
```

---

# 二十、最后用一句话总结

```text id="jcrenp"
Redis：让常用数据读得更快，让临时状态更好管理。
队列：让耗时任务排队后台处理，避免用户一直等待。
Nginx：作为系统入口，负责域名访问、HTTPS、路径转发和服务分流。
```

作为产品经理，你最重要的不是会配置它们，而是知道：

> **什么时候该用 Redis 加速，什么时候该用队列异步，什么时候问题可能出在 Nginx 转发或证书上。**

这三个能力掌握后，你和开发讨论性能、部署、接口、异步任务、环境隔离时，就会清楚很多。
