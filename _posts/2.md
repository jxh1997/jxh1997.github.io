# 阶段 1：软件系统全局认知

这是最基础的一层。你要先知道一个产品从用户点击到系统返回结果，中间到底发生了什么。

## 1.1 什么是前端、后端、数据库

你要理解：

```text
前端：用户能看到和操作的界面
后端：处理业务逻辑、权限、数据、规则的服务
数据库：存储业务数据的地方
服务器：运行后端服务的机器
接口：前端和后端沟通的通道
```

举例：

用户在小程序点击“搜索学校”。

完整链路是：

```text
用户点击搜索
↓
前端收集搜索条件
↓
调用后端搜索接口
↓
后端校验参数
↓
后端查询数据库
↓
数据库返回结果
↓
后端整理数据
↓
前端展示学校列表
```

你要形成一个意识：

> 页面上看到的每一个数据，背后都一定有来源。可能来自接口，接口来自后端，后端来自数据库或第三方系统。

---

## 1.2 什么是客户端和服务端

客户端包括：

```text
Web网页
APP
微信小程序
H5页面
后台管理系统
```

服务端包括：

```text
业务后端
数据库
文件服务
消息服务
第三方平台接口
IoT设备云平台
AI服务
支付服务
地图服务
```

产品经理要理解：

客户端不是系统全部，它只是入口。真正的业务规则通常在服务端。

比如：

```text
按钮是否可点击：前端可以控制
用户是否有权限：后端必须控制
订单是否能支付：后端判断
设备是否在线：后端根据设备上报数据判断
会员是否有效：后端根据会员到期时间判断
```

---

## 1.3 什么是“数据流”

你以后看任何功能，都要问：

```text
数据从哪里来？
经过谁处理？
存在哪里？
谁可以修改？
修改后影响哪里？
页面展示的是实时数据还是缓存数据？
```

比如设备在线状态：

```text
设备上报数据
↓
IoT平台接收
↓
后端记录最后上报时间
↓
系统根据离线规则判断在线/离线
↓
APP展示设备状态
```

这就是数据流。

---

## 1.4 什么是“业务流”

业务流是用户或业务对象的状态变化过程。

比如学校审核：

```text
待抓取
↓
抓取中
↓
抓取成功
↓
待翻译
↓
待审核
↓
审核通过
↓
已发布
```

比如订单：

```text
待支付
↓
已支付
↓
服务中
↓
已完成
↓
已评价
```

产品经理一定要把“页面”背后的“业务流”写清楚。

---

## 1.5 本阶段你要达到的能力

你看到一个功能时，能拆成：

```text
谁在用？
在哪个页面用？
点击了什么？
调用了什么接口？
数据从哪里来？
规则在哪里判断？
结果展示在哪里？
失败了怎么办？
```

这是所有后续学习的基础。

# 阶段 2：前端与页面技术

前端不是只负责“画页面”，它还负责用户交互、数据展示、页面状态、表单校验、加载体验、兼容适配。

## 2.1 页面结构

你要理解页面通常由这些组成：

```text
导航栏
筛选区
列表区
详情区
按钮区
弹窗
表单
分页
空状态
加载状态
错误状态
```

以后台设备列表为例：

```text
顶部：标题 + 新增按钮
筛选区：设备名称、设备SN、安装商、状态
列表区：设备数据表格
底部：分页器
弹窗：新增/编辑设备
```

产品经理写需求时，不要只写“做一个设备列表”，而要写清楚页面结构。

---

## 2.2 组件

组件就是可复用的页面模块。

常见组件：

```text
按钮
输入框
下拉框
表格
卡片
弹窗
日期选择器
上传组件
标签
步骤条
筛选器
分页器
```

你要理解：

如果多个页面都用同一类展示方式，开发通常会做成组件。

比如：

```text
设备状态标签
学校卡片
会员权益卡片
订单状态条
能流图组件
```

产品经理要关注组件的一致性：

```text
同一个状态颜色是否统一？
同一个按钮文案是否统一？
同一个筛选交互是否统一？
同一个空状态是否统一？
```

---

## 2.3 路由

路由就是页面地址和页面之间的跳转关系。

比如：

```text
/school/list        学校列表
/school/detail/123  学校详情
/order/list         订单列表
/order/detail/888   订单详情
```

小程序里也有页面路径：

```text
/pages/home/index
/pages/school/list
/pages/school/detail
/pages/user/profile
```

产品经理要在 PRD 里写清楚：

```text
从哪里进入？
跳到哪个页面？
是否带参数？
返回到哪里？
是否需要保留筛选条件？
```

比如：

用户从学校列表筛选“中国 + IB课程”，进入学校详情，再返回时，是否保留筛选条件？

这就是路由和页面状态问题。

---

## 2.4 页面状态

页面不是只有“正常展示”一种状态。

常见状态包括：

```text
加载中
加载成功
加载失败
无数据
无权限
网络异常
接口报错
部分数据为空
提交中
提交成功
提交失败
```

产品经理必须写这些状态。

比如学校列表：

```text
首次进入：显示骨架屏或加载中
有数据：显示学校列表
无数据：显示“暂无符合条件的学校”
接口失败：显示“加载失败，请重试”
无权限：显示“暂无访问权限”
```

如果 PRD 不写，开发可能随便处理，用户体验就会很差。

---

## 2.5 表单

表单是后台和小程序里最常见的东西。

你要理解表单的核心内容：

```text
字段名称
字段类型
是否必填
默认值
输入限制
校验规则
错误提示
提交接口
提交成功后去哪里
提交失败怎么提示
```

比如新增学校表单：

```text
学校名称：必填，最多100字
国家：必填，下拉选择
城市：非必填
课程体系：多选
学费范围：数字
官网链接：URL格式
Logo：图片上传
```

产品经理写表单需求时，要特别注意：

```text
字段从哪里来？
字段能不能为空？
字段有没有格式限制？
字段是单选还是多选？
字段修改后是否影响已发布数据？
```

---

## 2.6 前端校验与后端校验

这是你必须理解的知识点。

前端校验是为了提升体验。

比如：

```text
手机号格式不对，前端立即提示
必填项为空，前端不让提交
密码长度不足，前端提示
```

后端校验是为了保证安全和数据正确。

比如：

```text
用户是否有权限
订单金额是否正确
会员是否过期
设备是否属于当前用户
学校ID是否存在
```

产品经理要知道：

> 前端校验不能替代后端校验。

因为前端可以被绕过，后端才是真正可信的规则执行位置。

---

## 2.7 前端渲染

渲染就是把数据变成用户看到的页面。

你要理解：

```text
数据少，页面渲染快
数据多，页面渲染慢
图片多，加载慢
组件复杂，渲染慢
表格行数太多，浏览器卡顿
```

所以列表类页面要考虑：

```text
分页
虚拟列表
懒加载
骨架屏
图片压缩
接口分批加载
```

这不仅是性能问题，也是产品体验问题。

---

## 2.8 本阶段你要达到的能力

你能看懂一个页面，并拆出：

```text
页面结构
组件组成
页面跳转
加载状态
空状态
错误状态
表单校验
按钮权限
数据展示规则
```

# 阶段 3：接口与 HTTP

接口是产品经理最应该优先深入学习的技术内容。因为接口连接前端、后端、数据库、第三方系统。

## 3.1 什么是接口

接口就是系统之间约定好的数据交换方式。

比如：

前端想拿学校列表，就调用：

```text
GET /api/schools?page=1&pageSize=20&country=China
```

后端返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "total": 100,
    "list": []
  }
}
```

你要理解：

接口不是开发随便写的，它应该和业务需求严格对应。

---

## 3.2 请求方式：GET、POST、PUT、DELETE

常见请求方式：

```text
GET：查询数据
POST：新增数据或提交复杂操作
PUT：整体更新数据
PATCH：局部更新数据
DELETE：删除数据
```

产品经理要知道：

```text
查列表：GET
查详情：GET
新增：POST
编辑：PUT/PATCH
删除：DELETE
提交审核：POST
审核通过：POST
```

例如：

```text
GET /api/schools             查询学校列表
GET /api/schools/123         查询学校详情
POST /api/schools            新增学校
PUT /api/schools/123         编辑学校
DELETE /api/schools/123      删除学校
POST /api/schools/123/audit  审核学校
```

---

## 3.3 请求参数

请求参数是前端传给后端的条件。

常见参数类型：

```text
路径参数
查询参数
请求体参数
请求头参数
```

例子：

```text
路径参数：/api/schools/123 里的 123
查询参数：?page=1&pageSize=20
请求体参数：{"name":"xxx","country":"China"}
请求头参数：Authorization: Bearer token
```

产品经理要重点关注：

```text
哪些参数必传？
哪些参数可选？
默认值是什么？
参数格式是什么？
参数错误怎么提示？
参数是否影响权限？
```

---

## 3.4 返回数据

返回数据是后端给前端的结果。

一般包括：

```text
状态码
提示信息
业务数据
分页信息
错误信息
```

常见结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],
    "total": 100
  }
}
```

产品经理要看懂：

```text
页面每个字段来自返回数据的哪个字段？
如果字段为空，页面怎么展示？
如果接口没有这个字段，是前端不能展示，还是后端要补？
```

---

## 3.5 状态码

HTTP 状态码是接口请求结果的技术状态。

常见状态码：

```text
200：请求成功
400：参数错误
401：未登录
403：无权限
404：资源不存在
500：服务器错误
502：网关错误
504：超时
```

产品经理要能理解：

```text
401 通常跳登录
403 通常提示无权限
404 通常提示数据不存在
500 通常提示系统异常
504 通常提示请求超时
```

---

## 3.6 Token 和登录态

很多系统登录后会得到 token。

token 用来证明：

```text
你是谁
你是否登录
你有什么权限
```

前端每次请求接口时，通常会在请求头里带上 token。

```text
Authorization: Bearer xxxxx
```

产品经理要理解：

```text
未登录不能访问哪些页面？
token 过期后怎么处理？
是否自动刷新 token？
是否跳回登录页？
多端登录是否互踢？
```

---

## 3.7 分页接口

分页是列表类产品的基础能力。

常见请求：

```text
page=1
pageSize=20
```

返回：

```json
{
  "total": 300,
  "list": []
}
```

你要理解：

```text
page 表示第几页
pageSize 表示每页多少条
total 表示总数
list 表示当前页数据
```

产品经理要明确：

```text
默认每页多少条？
是否允许切换 pageSize？
搜索后是否回到第一页？
删除最后一条后页码怎么处理？
是否需要无限滚动？
```

---

## 3.8 接口幂等性

这是进阶但很重要的知识。

幂等性的意思是：

> 同一个操作执行一次和执行多次，结果应该一样。

比如：

```text
点击“确认支付”不能因为用户连点两次就创建两个订单。
点击“提交审核”不能重复生成多个审核任务。
点击“删除”重复请求不能导致异常数据。
```

产品经理要关注：

```text
按钮是否防重复点击？
后端是否防重复提交？
是否有唯一业务单号？
重复请求怎么返回？
```

这对订单、支付、审核、配网、设备绑定非常重要。

---

## 3.9 接口文档

接口文档不是开发自己的东西，产品经理也要能看。

你要能读懂：

```text
接口名称
接口地址
请求方式
请求参数
返回字段
错误码
示例数据
权限要求
```

你不用自己写代码，但要能判断：

```text
接口是否满足页面展示？
字段是否缺失？
参数是否支持筛选？
是否支持分页？
是否有权限控制？
错误提示是否清楚？
```

---

## 3.10 本阶段你要达到的能力

你能围绕一个页面输出：

```text
接口清单
每个接口的用途
请求参数
返回字段
错误状态
分页规则
权限要求
```

# 阶段 4：后端与业务逻辑

后端不是简单“查数据库”，它承担了真正的业务规则。

## 4.1 后端分层

常见后端结构：

```text
Controller：接收请求
Service：处理业务逻辑
DAO/Repository：访问数据库
Model/Entity：数据模型
Middleware：中间件，处理登录、权限、日志等
```

产品经理不用记代码，但要理解分工。

比如：

```text
Controller 接收“审核学校”的请求
Service 判断当前状态是否允许审核
DAO 查询学校数据
Service 修改审核状态
Controller 返回结果
```

---

## 4.2 业务逻辑在哪里

业务逻辑通常应该放在后端。

例如：

```text
会员是否过期
订单是否可以取消
设备是否属于当前用户
学校是否允许发布
电站状态如何计算
优惠券是否可用
```

这些不能只放在前端。

因为前端可以被绕过，后端才是可信规则层。

---

## 4.3 参数校验

后端要校验前端传来的参数。

比如：

```text
学校ID是否存在
订单金额是否合法
设备SN是否重复
用户是否有权限
状态是否允许流转
文件类型是否合法
```

产品经理要写清楚：

```text
什么情况下允许提交？
什么情况下不允许提交？
不允许时提示什么？
```

---

## 4.4 权限判断

后端要判断用户能不能操作。

常见权限：

```text
菜单权限
按钮权限
数据权限
接口权限
角色权限
组织权限
```

比如安装商平台：

```text
平台管理员：看全部电站
安装商管理员：看自己公司电站
普通安装人员：看自己负责的电站
C端用户：只看自己绑定的家庭电站
```

产品经理要把权限写到需求里，而不是只画页面。

---

## 4.5 同步任务与异步任务

同步任务是用户发起请求后立即等待结果。

比如：

```text
查询列表
保存表单
修改昵称
获取详情
```

异步任务是不适合让用户一直等的任务。

比如：

```text
批量导入
抓取学校详情
AI翻译
图片上传处理
生成报表
发送短信
设备配网结果同步
```

产品经理要理解：

如果一个任务耗时长，就不应该一直让页面转圈等待，而应该设计成：

```text
提交任务
↓
显示处理中
↓
后台异步执行
↓
用户稍后查看结果
↓
失败可重试
```

---

## 4.6 第三方系统对接

很多产品能力来自第三方系统。

比如：

```text
支付系统
短信系统
地图系统
IoT云平台
AI接口
微信登录
文件存储
ERP系统
CRM系统
```

对接第三方要关注：

```text
接口是否稳定？
调用是否有限频？
失败是否重试？
数据是否实时？
是否有回调？
是否有签名校验？
费用如何计算？
异常时怎么兜底？
```

产品经理必须知道：

> 第三方系统不受我们完全控制，所以一定要设计异常方案。

---

## 4.7 本阶段你要达到的能力

你能判断：

```text
哪些规则应该放前端？
哪些规则必须放后端？
哪些任务应该同步？
哪些任务应该异步？
哪些地方需要权限判断？
哪些地方依赖第三方？
```

# 阶段 5：数据库与数据建模

这是产品经理进阶非常关键的一层。很多需求不清楚，本质是数据关系没想清楚。

## 5.1 表

表是同一类数据的集合。

比如：

```text
用户表
学校表
设备表
电站表
订单表
会员表
审核记录表
```

产品经理要知道：

一个业务对象通常对应一张或多张表。

比如学校：

```text
学校基础信息表
学校图片表
学校课程表
学校审核表
学校版本表
学校发布记录表
```

---

## 5.2 字段

字段是表中的具体属性。

比如学校表字段：

```text
id
name
country
city
address
curriculum
age_min
age_max
fees_min
fees_max
status
created_at
updated_at
```

产品经理要关注：

```text
字段名称
字段含义
字段类型
是否必填
是否唯一
是否可修改
是否需要展示
是否参与筛选
是否参与排序
```

---

## 5.3 主键

主键是每条数据的唯一标识。

比如：

```text
学校ID
用户ID
订单ID
设备ID
```

产品经理要知道：

页面跳详情，通常需要带主键。

```text
点击学校A
↓
携带 school_id=123
↓
进入学校详情
```

主键不一定给用户看，但系统内部必须有。

---

## 5.4 外键和关联关系

外键表示两张表之间的关系。

比如：

```text
设备表里有 station_id，表示设备属于哪个电站
订单表里有 user_id，表示订单属于哪个用户
学校图片表里有 school_id，表示图片属于哪个学校
```

产品经理要理解这几种关系：

```text
一对一
一对多
多对多
```

例如：

```text
一个用户对应一个用户档案：一对一
一个电站有多个设备：一对多
一个学校有多个课程，一个课程也属于多个学校：多对多
```

---

## 5.5 枚举字段

枚举字段是固定范围内的状态或类型。

比如订单状态：

```text
pending_payment：待支付
paid：已支付
serving：服务中
completed：已完成
cancelled：已取消
```

设备状态：

```text
online：在线
offline：离线
fault：故障
unknown：未知
```

产品经理要把枚举写清楚：

```text
有哪些值？
每个值是什么意思？
前端显示什么文案？
每个状态能做什么操作？
状态如何变化？
```

---

## 5.6 数据冗余

正常情况下，数据不应该重复存太多。

但有时为了查询性能或历史记录，需要冗余。

比如订单里保存商品名称：

```text
商品表里有商品名称
订单表里也保存下单时的商品名称
```

为什么？

因为商品后续可能改名，但历史订单要保留当时名称。

产品经理要理解：

```text
哪些数据要实时读取？
哪些数据要保留历史快照？
哪些数据允许冗余？
哪些数据必须保持一致？
```

---

## 5.7 数据快照

快照是某一时刻的数据副本。

常见场景：

```text
订单下单时的商品信息
学校发布时的审核版本
用户购买会员时的权益信息
合同签署时的客户信息
```

比如学校信息审核：

```text
原始抓取数据
AI翻译数据
人工审核数据
最终发布数据
```

这些都可能需要保存版本，而不是只保留最新结果。

---

## 5.8 事务

事务保证一组数据库操作要么全部成功，要么全部失败。

比如用户支付成功后：

```text
创建支付记录
更新订单状态
开通会员
写入会员权益
发送通知
```

其中关键数据库操作不能成功一半失败一半。

产品经理要知道：

涉及钱、权益、库存、订单、设备绑定的地方，通常要考虑事务。

---

## 5.9 本阶段你要达到的能力

你能围绕一个业务对象拆出：

```text
需要哪些表？
每张表存什么？
表之间是什么关系？
哪些字段必填？
哪些字段唯一？
哪些字段是状态？
哪些数据需要留历史？
哪些操作要保证一致性？
```

# 阶段 6：查询、索引与性能

这个阶段不只讲“加载慢”，而是讲系统如何高效处理数据。

## 6.1 查询

查询就是从数据库里找数据。

常见查询方式：

```text
按ID查询
按名称查询
按状态查询
按时间范围查询
按多个条件组合查询
模糊查询
排序查询
分页查询
联表查询
```

产品经理要理解：

查询条件越多，不代表一定慢。真正影响速度的是：

```text
数据量
索引
查询方式
排序方式
联表复杂度
返回数据量
数据库设计
```

---

## 6.2 精确查询和模糊查询

精确查询：

```text
设备SN = ABC123
订单号 = 202607090001
手机号 = 138xxxx
```

模糊查询：

```text
学校名称包含“International”
设备名称包含“逆变器”
客户名称包含“张”
```

模糊查询通常更慢，尤其是前后都模糊：

```sql
LIKE '%关键词%'
```

产品经理要知道：

```text
精确查询适合编号、SN、手机号、订单号
模糊查询适合名称、备注、描述
大数据量下模糊查询需要谨慎
```

---

## 6.3 索引

索引可以理解为数据库的目录。

没有索引时：

```text
数据库可能要一条一条找
```

有索引时：

```text
数据库可以通过目录快速定位
```

适合加索引的字段：

```text
经常用于筛选的字段
经常用于排序的字段
经常用于关联的字段
唯一字段
```

比如：

```text
设备SN
用户ID
电站ID
订单号
状态
创建时间
安装商ID
```

但索引不是越多越好。

因为索引会增加写入和维护成本。

产品经理要理解：

> 高频查询字段才值得考虑索引。

---

## 6.4 分页

分页是控制一次返回数据量。

没有分页：

```text
一次返回 90000 条设备
前端卡
接口慢
数据库压力大
网络传输慢
```

有分页：

```text
每次返回 20 条或 50 条
接口更快
页面更流畅
用户更容易浏览
```

产品经理要考虑：

```text
默认每页多少条？
是否支持切换每页条数？
搜索后是否回到第一页？
分页是否显示总数？
是否使用无限滚动？
```

---

## 6.5 排序

排序也会影响性能。

常见排序：

```text
按创建时间倒序
按更新时间倒序
按金额排序
按状态排序
按名称排序
```

大数据量下，如果排序字段没有索引，也可能变慢。

产品经理要关注：

```text
默认排序是什么？
用户是否可以手动排序？
排序字段是否固定？
排序是否影响业务优先级？
```

---

## 6.6 缓存

缓存就是把经常用的数据临时存起来，避免每次都重新查询。

适合缓存的数据：

```text
国家地区列表
枚举字典
首页配置
热门内容
用户权限菜单
设备统计汇总
学校筛选项
```

不适合长期缓存的数据：

```text
订单支付状态
设备实时状态
账户余额
库存数量
强实时数据
```

产品经理要理解缓存的一个核心问题：

> 缓存会提升速度，但可能带来数据延迟。

所以需要明确：

```text
缓存多久？
什么时候刷新？
用户能否接受延迟？
是否需要手动刷新？
```

---

## 6.7 本阶段你要达到的能力

你能判断：

```text
哪些字段适合查询？
哪些字段适合索引？
哪些数据需要分页？
哪些数据适合缓存？
哪些查询可能慢？
哪些页面需要加载优化？
```

# 阶段 7：业务规则与状态机

这个阶段是产品经理必须深入掌握的核心能力。

## 7.1 状态

状态表示一个业务对象当前处于什么阶段。

比如订单状态：

```text
待支付
已支付
服务中
已完成
已取消
退款中
已退款
```

学校状态：

```text
待抓取
抓取中
待审核
审核通过
已发布
已下架
```

设备状态：

```text
在线
离线
故障
待配网
配网中
配网失败
```

产品经理要写清楚每个状态的含义。

---

## 7.2 状态流转

状态流转表示状态如何变化。

比如订单：

```text
待支付 → 已支付
待支付 → 已取消
已支付 → 服务中
服务中 → 已完成
已支付 → 退款中
退款中 → 已退款
```

你不能只写状态，还要写：

```text
谁触发？
什么条件下触发？
触发后变成什么状态？
失败怎么办？
是否允许回退？
```

---

## 7.3 状态和按钮权限

不同状态下，可操作按钮不同。

比如学校审核：

| 状态   | 可操作        |
| ---- | ---------- |
| 待审核  | 通过、驳回、编辑   |
| 已发布  | 下架、编辑、重新发布 |
| 已下架  | 上架、编辑      |
| 抓取中  | 查看进度       |
| 抓取失败 | 重新抓取       |

产品经理要把状态和操作关联起来。

---

## 7.4 异常状态

很多 PRD 最大的问题是只写正常流程，不写异常流程。

异常状态包括：

```text
支付失败
审核驳回
配网失败
设备离线
接口超时
文件上传失败
AI生成失败
第三方系统无响应
订单取消
库存不足
权限变更
```

产品经理必须写：

```text
异常出现后，用户看到什么？
是否可以重试？
是否需要人工处理？
是否通知用户？
是否记录日志？
是否影响主流程？
```

---

## 7.5 状态机

状态机就是把所有状态、流转、触发条件整理成一套规则。

一个完整状态机要包括：

```text
状态列表
初始状态
终态
允许的流转
触发动作
前置条件
失败处理
操作权限
页面展示
```

比如设备配网：

```text
未配网
↓ 用户开始配网
配网中
↓ 成功
已联网
↓ 绑定家庭
已绑定
```

异常流：

```text
配网中
↓ 超时
配网失败
↓ 用户重试
配网中
```

---

## 7.6 本阶段你要达到的能力

你能把任何复杂业务拆成：

```text
状态有哪些？
每个状态是什么意思？
状态怎么流转？
谁能触发？
触发条件是什么？
失败怎么办？
每个状态页面展示什么？
每个状态能操作什么？
```

# 阶段 8：权限、安全与账号体系

权限是后台、B端、C端系统都绕不开的问题。

## 8.1 用户、角色、权限

三者关系：

```text
用户：具体的人
角色：一类人
权限：能看什么、能做什么
```

比如：

```text
用户：张三
角色：安装商管理员
权限：查看电站、创建设备、编辑设备、查看统计
```

---

## 8.2 菜单权限

控制用户能看到哪些菜单。

比如：

```text
平台管理员：全部菜单
运营人员：内容管理、学校审核
财务人员：订单、支付、退款
安装商：电站管理、设备管理
普通用户：没有后台权限
```

---

## 8.3 按钮权限

控制用户能不能操作。

比如：

```text
查看
新增
编辑
删除
审核
导出
发布
下架
退款
重试
```

同一个页面，不同角色看到的按钮可能不同。

---

## 8.4 数据权限

这是最容易被产品经理忽略的。

数据权限控制用户能看到哪些数据。

比如：

```text
平台管理员：看全部数据
区域管理员：看某个区域数据
安装商：看自己名下电站
安装人员：看自己负责的安装任务
C端用户：看自己绑定的家庭
```

如果数据权限没设计好，会产生严重问题。

比如安装商 A 看到了安装商 B 的客户数据，就是权限事故。

---

## 8.5 登录态和会话

你要理解：

```text
登录
退出登录
token过期
自动续期
多端登录
账号冻结
密码重置
微信授权
手机号绑定
```

产品经理要明确：

```text
多久登录过期？
过期后是否跳登录？
是否支持多端登录？
修改密码后是否踢出其他设备？
账号禁用后正在登录的人怎么办？
```

---

## 8.6 本阶段你要达到的能力

你能设计：

```text
角色列表
菜单权限
按钮权限
数据权限
登录过期规则
账号禁用规则
权限异常提示
```

# 阶段 9：系统架构与基础设施

这部分不用学到运维级别，但必须知道每个组件解决什么问题。

## 9.1 服务器

服务器就是运行系统的机器。

你要知道：

```text
前端文件可以部署在服务器
后端服务运行在服务器
数据库可以部署在服务器
Nginx 可以部署在服务器
```

产品经理需要理解：

```text
测试环境
预发布环境
生产环境
```

测试环境用于开发测试，生产环境给真实用户用。

---

## 9.2 Nginx

Nginx 常用来做：

```text
反向代理
静态资源服务
HTTPS证书配置
域名转发
负载均衡
路径转发
HTTP跳HTTPS
```

比如：

```text
/api → 转发到后端服务
/admin → 返回后台页面
/school-intelligence → 返回另一个系统页面
```

产品经理不需要会写完整配置，但要知道：

> 域名、路径、证书、接口转发，经常和 Nginx 有关。

---

## 9.3 PM2

PM2 是 Node 服务常用的进程管理工具。

它解决：

```text
启动服务
停止服务
重启服务
查看日志
服务崩溃后自动拉起
```

如果你们后端是 Node，PM2 很常见。

产品经理要知道：

```text
服务是否启动
服务是否重启
日志在哪里看
异常是否频繁出现
```

---

## 9.4 文件存储 COS/OSS

图片、PDF、视频、附件，一般不建议长期存在业务服务器本地。

通常会上传到对象存储：

```text
腾讯云 COS
阿里云 OSS
AWS S3
```

适合存：

```text
学校图片
用户头像
合同文件
PDF资料
设备日志文件
报表文件
```

产品经理要关注：

```text
文件大小限制
文件类型限制
上传失败处理
访问权限
是否需要防盗链
是否需要压缩
是否需要缩略图
```

---

## 9.5 Redis

Redis 常用于：

```text
缓存
验证码
登录态
分布式锁
排行榜
临时数据
队列辅助
```

产品经理需要理解几个场景：

```text
热门数据加缓存
验证码5分钟过期
防止重复提交
限制接口频繁调用
```

不要把 Redis 简单理解成“更快的数据库”。它更像高速临时存储。

---

## 9.6 消息队列

队列用来处理异步任务。

适合：

```text
发送短信
发送邮件
图片处理
AI翻译
批量导入
学校抓取
生成报表
设备数据清洗
```

队列的好处：

```text
不阻塞用户操作
失败可以重试
任务可以排队
可以削峰填谷
```

产品经理要理解：

如果任务不是马上要结果，就可以考虑异步队列。

---

## 9.7 日志

日志是系统排查问题的证据。

常见日志：

```text
接口访问日志
错误日志
登录日志
操作日志
任务执行日志
设备上报日志
第三方接口日志
```

产品经理要知道：

没有日志，很多问题只能猜。

比如用户说“我刚才提交失败了”，如果有日志，可以查：

```text
用户是谁？
什么时候提交？
调用哪个接口？
参数是什么？
返回什么错误？
后端有没有异常？
第三方有没有失败？
```

---

## 9.8 本阶段你要达到的能力

你能听懂：

```text
服务挂了
Nginx转发错了
PM2进程没启动
接口502
证书过期
Redis缓存没刷新
队列积压
COS上传失败
日志没有记录
```

# 阶段 10：IoT、AI、内容系统等业务技术扩展

## 10.2 能源数据

家庭能源管理要理解：

```text

负载功率

充电量
放电量

收益

能流图
工作模式
```

你要区分：

```text

```

比如：

```text

```

这两个不是一回事。

# 每个知识点的学习深度标准

你以后学任何一个技术点，都要达到这 5 层。

## 第一层：知道它是什么

比如缓存：

```text
缓存是把常用数据临时存起来，减少重复查询。
```

## 第二层：知道为什么需要它

```text
因为有些数据访问频繁，每次查数据库会慢，也会增加数据库压力。
```

## 第三层：知道适合什么场景

```text
国家列表、枚举字典、首页配置、热门内容适合缓存。
```

## 第四层：知道有什么风险

```text
缓存可能导致数据不是最新的，需要考虑过期时间和刷新机制。
```

## 第五层：知道产品需求怎么写

```text
国家地区筛选项允许缓存，缓存时间 24 小时；
后台修改国家地区后，支持手动刷新缓存；
若缓存加载失败，降级为实时查询数据库。
```

这才叫真正学懂。
