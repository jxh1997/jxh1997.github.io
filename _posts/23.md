请用产品经理能理解的方式，深入讲解【菜单权限是怎么实现的】：

你图里的页面，本质是在做一件事：

> **给某个角色配置它能看到哪些菜单、能点哪些按钮、能访问哪些 APP 功能，以及它能看哪些范围的数据。**

你这个截图里已经包含了几类权限：

```text
1. 基本信息：角色名称、角色类型
2. 数据权限：全部数据权限 / 本部门 / 本人 / 自定义等
3. Web菜单权限：后台管理端能看到哪些菜单、按钮
4. APP菜单权限：B端 APP 能看到哪些功能入口
```

下面我按你给的 9 个维度详细讲。

---

# 1. 菜单权限是什么？

菜单权限就是：

> **系统根据用户角色，决定这个用户登录后能看到哪些菜单、页面、按钮和功能入口。**

比如同一个 HaierEnergy 后台：

平台管理员登录后能看到：

```text
电站管理
设备管理
任务中心
远程升级
组织管理
用户管理
系统管理
报表中心
```

安装商管理员登录后可能只能看到：

```text
电站管理
设备管理
任务中心
远程升级
```

普通安装工可能只能看到：

```text
我的电站
我的设备
任务中心
```

所以菜单权限解决的是：

> **不同角色进入同一个系统，看到的功能范围不同。**

---

# 2. 菜单权限不只是“左侧菜单”

很多产品经理一听“菜单权限”，会以为只是控制左侧导航栏显示不显示。

其实菜单权限通常分三层。

---

## 第一层：目录权限

也就是一级菜单、二级菜单能不能看到。

比如：

```text
设备管理
远程升级
系统管理
报表中心
```

如果用户没有“系统管理”权限，左侧导航就不显示“系统管理”。

---

## 第二层：页面权限

用户能不能进入某个具体页面。

比如：

```text
设备列表页
设备详情页
角色管理页
权限管理页
OTA任务详情页
```

有些页面不一定出现在左侧菜单里，但用户仍可能通过链接进入。

比如你截图里的“详情”抽屉页，可能不是菜单，但它也是一个页面能力。

---

## 第三层：按钮 / 操作权限

页面进来了之后，还要控制能不能操作。

比如设备管理页里：

```text
新增设备
删除设备
编辑设备
导出设备
远程升级
下发指令
查看详情
```

你的截图里能看到类似：

```text
删除设备
设备更名
设备管理-设备SN点击
设备管理-电站点击
新增设备
实时监控-读取/刷新
实时监控-下发
实时监控-历史
实时监控-列表导出
```

这些就不是单纯菜单了，而是**按钮权限 / 操作权限**。

---

# 3. 为什么需要菜单权限？

因为后台系统不是给一个人用的。

HaierEnergy 这类系统会有很多角色：

```text
平台管理员
总部人员
区域管理员
安装商管理员
安装工
售后人员
运营人员
客服人员
财务人员
客户用户
子公司人员
```

这些人不能看到完全一样的功能。

---

## 举例

安装工不应该看到：

```text
角色管理
权限管理
组织管理
系统配置
全部用户数据
```

客服可能需要看设备详情，但不应该操作：

```text
删除设备
远程升级
下发控制指令
```

运营人员可能能看报表，但不应该能：

```text
新增设备
修改电站
解绑设备
```

所以菜单权限的目的不是“页面少一点”，而是：

> **让不同岗位只拥有完成工作所需的最小功能范围。**

---

# 4. 它解决什么问题？

菜单权限主要解决 6 类问题。

---

## 4.1 防止用户看到不该看的功能

比如安装工登录后台，不应该看到：

```text
系统管理
角色管理
权限管理
数据大屏配置
```

否则用户会困惑，也有安全风险。

---

## 4.2 防止用户误操作

比如一个普通安装工如果能看到“删除设备”，可能误删设备。

所以可以让他：

```text
能看设备
不能删除设备
不能远程升级
不能下发控制
```

---

## 4.3 支持组织分工

不同部门用同一套系统，但功能不同。

例如：

```text
运维部门：看设备、告警、OTA
安装商：建站、添加设备、交付用户
客服：查看用户和电站状态
管理员：配置角色、权限、字典、多语言
```

---

## 4.4 支持 B 端和 Web 后台差异

你截图里已经区分了：

```text
Web菜单权限
APP菜单权限
```

这是对的。

因为后台 Web 和 B端 APP 功能不一样。

比如：

Web 后台有：

```text
角色管理
权限管理
报表中心
固件管理
```

B端 APP 有：

```text
交付用户
新增电站
设备扫码
现场配网
```

所以权限不能只设计一套“后台菜单”，要区分终端。

---

## 4.5 支持数据安全

菜单权限控制“能不能进入功能”。

数据权限控制“能看哪些数据”。

比如两个人都有“设备管理”菜单权限。

但：

```text
总部管理员：能看全部设备
安装商管理员：只能看本安装商设备
安装工：只能看自己负责的设备
```

这就是菜单权限 + 数据权限共同作用。

---

## 4.6 支持后续扩展

以后 HaierEnergy 上新功能，比如：

```text
OTA升级
电价策略
虚拟电厂
能源报表
澳洲数据中心
欧洲动态电价
```

不应该默认所有角色都能看到。

新功能上线后，需要通过权限配置决定谁可见、谁可操作。

# 5. 产品经理在需求中怎么体现？

你不能只写：

```text
支持角色权限管理。
```

这太模糊。

应该拆成下面几部分。

---

## 5.1 角色管理

先定义角色。

比如：

| 角色      | 说明               |
| ------- | ---------------- |
| 平台超级管理员 | 拥有全部功能和全部数据      |
| 平台运维    | 管理设备、告警、OTA      |
| 安装商管理员  | 管理本安装商下的用户、电站、设备 |
| 安装工     | 负责现场建站、配网、交付     |
| 客服      | 查看用户、电站、设备状态     |
| 只读角色    | 只能查看，不能修改        |

你截图里的角色类型有：

```text
安装商管理员
平台角色
安装商角色
```

这说明你们已经在做角色分层了。

---

## 5.2 菜单权限范围

要明确权限控制到哪一级。

例如：

```text
一级菜单
二级菜单
页面
按钮
接口
APP功能入口
```

PRD 写法：

```text
系统支持按角色配置 Web 后台菜单权限、Web 按钮权限、B端 APP 菜单权限。用户登录后，仅展示其角色拥有权限的菜单和功能入口。
```

---

## 5.3 按钮权限

按钮权限要单独列出来。

比如设备管理页面：

| 功能点  | 权限编码          | 说明        |
| ---- | ------------- | --------- |
| 新增设备 | device:add    | 控制新增按钮    |
| 删除设备 | device:delete | 控制删除按钮    |
| 编辑设备 | device:edit   | 控制编辑按钮    |
| 设备详情 | device:detail | 控制详情入口    |
| 远程升级 | device:ota    | 控制 OTA 操作 |
| 导出设备 | device:export | 控制导出按钮    |

你图里这种：

```text
删除设备
设备更名
新增设备
实时监控-下发
实时监控-列表导出
```

都应该是按钮/操作权限。

---

## 5.4 数据权限

你图里有：

```text
数据权限：全部数据权限
```

这一块非常重要。

产品经理要定义数据范围：

```text
全部数据
本组织数据
本部门数据
本部门及子部门数据
本人数据
自定义组织数据
自定义安装商数据
```

在 HaierEnergy 场景里，建议数据权限至少要覆盖：

```text
公司
部门
子公司
安装商
电站
设备
用户
区域
国家
```

比如：

```text
安装商管理员：可查看本安装商及下级安装工创建的电站和设备。
安装工：仅可查看本人创建或分配给本人的电站和设备。
平台运维：可查看全部电站和设备。
```

---

## 5.5 Web 和 APP 权限分开

你截图里已经有：

```text
web菜单权限
APP菜单权限
```

这是合理的。

PRD 可以写：

```text
角色权限需区分 Web 端和 APP 端。Web 端菜单权限用于控制后台管理系统菜单、页面和按钮；APP 端菜单权限用于控制 B端 APP 功能入口和操作能力。
```

例如：

同一个“安装商管理员”：

Web 端可能有：

```text
电站管理
设备管理
用户管理
任务中心
```

APP 端可能有：

```text
新增电站
扫码配网
交付用户
现场调试
```

---

## 5.6 权限生效规则

产品上必须定义：

```text
权限修改后是否立即生效？
用户是否需要重新登录？
已打开页面是否刷新？
Token 中是否缓存权限？
```

推荐写法：

```text
角色权限修改后，对新登录用户立即生效；已登录用户在刷新页面或重新登录后生效。高风险权限变更，如取消远程升级、删除设备权限，应在下次接口请求时立即校验生效。
```

注意最后一句很重要。

因为真正安全不能只靠前端刷新菜单。

# 6. 开发通常怎么实现？

菜单权限通常是 RBAC 模型。

RBAC 是：

```text
Role-Based Access Control
基于角色的访问控制
```

大白话就是：

> 用户不直接配置权限，而是给用户分配角色，角色再拥有权限。

---

## 6.1 基础关系

核心关系是：

```text
用户 → 角色 → 权限
```

例如：

```text
张三
↓
安装商管理员
↓
设备管理、新增设备、交付用户、查看电站
```

如果一个用户可以有多个角色，就变成：

```text
用户 → 多个角色 → 合并权限
```

---

## 6.2 常见数据库表

开发通常会有这些表。

### 用户表

```text
user
- id
- username
- phone
- org_id
```

### 角色表

```text
role
- id
- role_name
- role_type
- data_scope
```

### 菜单表

```text
menu
- id
- parent_id
- name
- path
- component
- type
- permission_code
- platform
- sort
```

这里的 type 很关键。

通常会有：

```text
M：目录
C：菜单
F：按钮
```

或者：

```text
directory
menu
button
api
```

### 用户角色关系表

```text
user_role
- user_id
- role_id
```

### 角色权限关系表

```text
role_menu
- role_id
- menu_id
```

如果要细分 APP 和 Web，可以在 menu 表里加：

```text
platform = web / app
```

或者拆成：

```text
web_menu
app_menu
```

---

## 6.3 登录后怎么返回菜单？

用户登录后，后端会根据用户角色查权限。

流程：

```text
用户登录
↓
后端查询用户角色
↓
后端查询角色拥有的菜单权限
↓
生成菜单树
↓
返回给前端
↓
前端渲染左侧菜单
```

比如后端返回：

```json
[
  {
    "name": "设备管理",
    "path": "/device",
    "children": [
      {
        "name": "设备列表",
        "path": "/device/list"
      }
    ]
  }
]
```

前端拿到后，只渲染这些菜单。

---

## 6.4 按钮权限怎么控制？

按钮权限通常靠权限编码。

例如用户拥有：

```text
device:add
device:edit
device:delete
device:export
```

前端判断：

```text
有 device:add → 显示新增按钮
没有 device:delete → 不显示删除按钮
```

但是这里有一个重点：

> **前端隐藏按钮只是用户体验，后端接口仍然必须校验权限。**

否则用户可以绕过页面，直接调用接口。

---

## 6.5 接口权限怎么控制？

比如删除设备接口：

```text
DELETE /api/device/{id}
```

后端要校验当前用户是否有：

```text
device:delete
```

没有权限就返回：

```text
403 Forbidden
```

这才是真正安全。

完整流程：

```text
用户点击删除设备
↓
前端判断是否有 delete 按钮权限
↓
有权限才显示按钮
↓
用户点击按钮
↓
请求后端删除接口
↓
后端再次校验 device:delete
↓
校验通过才执行删除
```

---

## 6.6 数据权限怎么实现？

菜单权限控制“能不能进”。

数据权限控制“能看到哪些数据”。

比如设备列表接口：

```text
GET /api/device/list
```

同一个接口，不同角色返回的数据不同。

平台管理员：

```sql
SELECT * FROM device
```

安装商管理员：

```sql
SELECT * FROM device WHERE installer_id = 当前安装商ID
```

安装工：

```sql
SELECT * FROM device WHERE creator_id = 当前用户ID
```

所以数据权限通常会在后端查询时追加条件。

# 7. 常见错误理解

## 错误 1：隐藏菜单就等于有权限控制

错。

隐藏菜单只是前端展示控制。

真正权限必须在后端接口校验。

否则用户可以通过接口地址直接访问。

---

## 错误 2：有菜单权限就一定能操作

不一定。

用户可能能进入设备管理页面，但不能删除设备。

所以要区分：

```text
菜单权限
按钮权限
接口权限
数据权限
```

---

## 错误 3：有设备管理权限就能看全部设备

不一定。

设备管理是菜单权限。

看哪些设备是数据权限。

比如安装工可以有设备管理菜单，但只能看自己负责的设备。

---

## 错误 4：角色越多越好

不对。

角色太多会导致管理混乱。

比如：

```text
安装工A角色
安装工B角色
安装工C角色
```

这不合理。

应该是：

```text
安装工角色 + 数据范围区分
```

不要用角色硬堆数据范围。

---

## 错误 5：权限只在后台管理系统需要

不对。

APP 也需要权限。

你图里有 APP 菜单权限，这是正确方向。

比如 B端 APP：

```text
有的人能新增电站
有的人只能查看电站
有的人能交付用户
有的人能扫码配网
有的人能远程升级
```

# 8. 真实业务案例：HaierEnergy 安装商管理员

结合你截图里的“测试回归 / 安装商管理员”。

假设这个角色是安装商管理员。

他应该拥有：

## Web 菜单权限

```text

设备管理

```

但不应该拥有：

```text
角色管理
权限管理
系统配置

```

---

## 设备管理按钮权限

可以：

```text

```

谨慎开放：

```text

```

因为这些操作风险更高。

---

## APP 菜单权限

可以：

```text
B端APP-新增电站
B端APP-交付用户
B端APP-扫码配网
B端APP-查看设备
```

---

## 数据权限

安装商管理员通常应该是：

```text
本安装商及下级人员数据
```

而不是：

```text
全部数据权限
```

你截图里“测试回归”是“全部数据权限”，这可能是测试角色可以接受。

但正式安装商管理员一般不建议给全部数据。

# 9. 你应该怎么问开发？

你可以直接问这几组问题。

---

## 9.1 问权限模型

```text
当前系统是 RBAC 权限模型吗？
用户和角色是一对一还是多对多？
角色和菜单权限是一对多还是多对多？
Web 菜单和 APP 菜单是否共用一套权限表？
按钮权限是否也在菜单表里管理？
```

---

## 9.2 问菜单权限

```text
菜单权限控制到几级？
一级菜单、二级菜单、页面、按钮是否都支持？
没有菜单权限时，前端是不显示，还是访问时提示无权限？
如果用户手动输入 URL，后端是否拦截？
```

---

## 9.3 问按钮权限

```text
按钮权限是否有独立 permission_code？
前端隐藏按钮，后端接口是否也校验？
比如没有 device:delete 权限，直接调用删除接口会不会被拦截？
```

---

## 9.4 问数据权限

```text
数据权限有哪些范围？
全部数据、本部门、本部门及子部门、本人、自定义是否支持？
安装商管理员的数据范围按什么字段过滤？
安装工只能看自己创建的电站，还是分配给自己的电站？
```

---

## 9.5 问权限生效

```text
修改角色权限后是否立即生效？
已登录用户是否需要重新登录？
Token 里是否缓存了权限？
权限缓存存在 Redis 里吗？
缓存多久刷新？
```

---

## 9.6 问多端权限

```text
Web 和 APP 权限是否分开配置？
同一个角色能不能配置不同端权限？
APP 功能入口隐藏后，APP 接口是否仍然校验权限？
```

# 10. PRD 示例写法

你可以这样写角色权限需求。

```text
功能名称：角色菜单权限管理

一、功能目标
系统支持按角色配置 Web 端菜单权限、Web 端按钮权限、APP 端菜单权限及数据权限。用户登录后，仅展示其角色拥有权限的菜单和功能入口。

二、权限类型
权限分为：
1. 菜单权限：控制用户是否可见某个菜单或页面；
2. 按钮权限：控制用户是否可执行新增、编辑、删除、导出、下发、远程升级等操作；
3. APP 权限：控制 B端 APP 功能入口及操作能力；
4. 数据权限：控制用户可查看和操作的数据范围。

三、权限配置
管理员可在角色详情页配置权限。
权限以树形结构展示，支持父子级联勾选。
取消父级权限时，默认取消其下级权限。
允许单独勾选子级按钮权限，但前提是用户具备对应页面访问权限。

四、权限生效
用户登录后，系统根据其角色返回对应菜单树和按钮权限集合。
用户无权限的菜单不展示。
用户无权限的按钮不展示。
用户直接访问无权限页面时，系统提示“暂无访问权限”。
用户调用无权限接口时，后端返回 403。

五、数据权限
数据权限支持：全部数据、本组织数据、本部门及子部门数据、本人数据、自定义数据范围。
设备、电站、用户、任务、报表等列表接口需按用户数据权限过滤结果。

六、权限修改
角色权限修改后，新登录用户立即生效。
已登录用户刷新页面后更新菜单权限。
高风险操作接口以服务端实时权限校验为准。
```

# 最后总结

菜单权限不是简单“左侧菜单显示隐藏”。

完整权限体系应该是：

```text
菜单权限：能不能看到功能入口
页面权限：能不能进入页面
按钮权限：能不能执行操作
接口权限：后端是否允许执行
数据权限：能操作哪些范围的数据
APP权限：移动端能看到哪些功能
```

对你图里的页面来说，最核心的产品判断是：

> **角色权限配置页不仅要控制菜单显示，还要和按钮权限、接口权限、数据权限联动，否则只是“看起来控制了权限”，实际并不安全。**
