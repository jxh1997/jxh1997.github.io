你问的是这句：

> 用户从学校列表筛选“中国 + IB课程”，进入学校详情，再返回时，是否保留筛选条件？
> 这就是路由和页面状态问题。

我换一种更直白的方式解释。

---

# 1. 路由是什么？

**路由 = 当前页面在哪里。**

你可以把路由理解成“页面地址”。

比如：

```text
/pages/school/list
```

表示学校列表页。

```text
/pages/school/detail?id=123
```

表示学校详情页，并且详情页要查看 ID 为 123 的学校。

所以路由主要解决：

```text
用户现在在哪个页面？
从哪个页面跳到哪个页面？
跳转时要不要带参数？
```

---

# 2. 页面状态是什么？

**页面状态 = 当前页面里的临时操作结果。**

比如你在学校列表页做了这些操作：

```text
选择国家：中国
选择课程：IB
输入关键词：国际学校
当前页码：第 3 页
排序方式：按学费从低到高
滚动位置：列表中间
```

这些不一定是一个新页面，但它们会影响当前页面展示什么内容。

这就叫页面状态。

---

# 3. 路由和页面状态的关系

路由负责“你在哪个页面”。

页面状态负责“这个页面当前是什么样子”。

比如：

```text
路由：/pages/school/list
页面状态：
- 国家 = 中国
- 课程 = IB
- 页码 = 3
- 关键词 = 国际学校
```

也就是说，同样是学校列表页，不同状态下看到的内容可能完全不同。

---

# 4. 为什么“进入详情再返回”会涉及这个问题？

假设用户在学校列表页做了筛选：

```text
国家：中国
课程：IB
页码：第 3 页
```

然后点击某个学校进入详情页：

```text
/pages/school/detail?id=123
```

用户看完详情后点返回。

这时产品经理要定义：

## 情况 A：返回后保留原来的筛选条件

返回列表页后，用户仍然看到：

```text
国家：中国
课程：IB
页码：第 3 页
```

用户体验比较好，因为他可以继续看刚才筛选出来的学校。

这叫：

> 保留页面状态。

---

## 情况 B：返回后列表重置

返回列表页后，页面变成：

```text
国家：全部
课程：全部
页码：第 1 页
```

用户刚才筛选的条件全部没了。

这叫：

> 页面状态丢失。

用户会觉得很烦，因为他还要重新筛选一次。

---

# 5. 所以这个问题的本质是什么？

本质是：

> 页面跳转时，之前页面的筛选条件、页码、滚动位置，要不要保留。

这就是“路由和页面状态”的关系。

---

# 6. 用你的产品场景举例

比如后台设备列表页。

用户筛选：

```text
安装商：A公司
设备状态：离线
建设状态：已建站
当前页：第 5 页
```

然后点进某个设备详情。

看完后返回设备列表。

你要决定：

## 推荐做法

返回后应该保留：

```text
安装商：A公司
设备状态：离线
建设状态：已建站
当前页：第 5 页
```

否则用户会很痛苦。

因为他本来是在一批筛选结果里逐个查看设备，返回后如果重置，相当于打断了他的工作流。

---

# 7. 哪些页面状态应该保留？

一般建议保留这些：

```text
筛选条件
搜索关键词
当前页码
排序方式
Tab选中项
滚动位置
列表展开状态
```

比如：

```text
学校列表 → 学校详情 → 返回学校列表
设备列表 → 设备详情 → 返回设备列表
订单列表 → 订单详情 → 返回订单列表
内容列表 → 文章详情 → 返回内容列表
```

这些场景都建议保留列表状态。

---

# 8. 哪些页面状态不一定要保留？

有些状态不建议保留，比如：

```text
弹窗打开状态
提交中的 loading 状态
临时错误提示
验证码倒计时之外的短暂提示
一次性操作成功提示
```

比如用户打开了“新增设备弹窗”，然后离开页面，回来时一般不应该还保持弹窗打开。

---

# 9. 产品需求里应该怎么写？

你在 PRD 里可以这样写：

```text
用户从学校列表页进入学校详情页后，点击返回时，应回到进入详情前的列表状态，包括筛选条件、搜索关键词、排序方式、当前页码和滚动位置。
```

或者设备列表：

```text
用户从设备列表页进入设备详情页后，返回设备列表时，应保留原筛选条件、当前页码和列表滚动位置，避免用户重复筛选。
```

这就是产品经理需要补充的交互细节。

---

# 10. 开发上通常怎么实现？

你不需要写代码，但要知道开发大概有几种方式：

```text
1. 把筛选条件放在 URL 参数里
2. 把状态存在页面缓存里
3. 把状态存在全局状态管理里
4. 小程序里使用页面栈保留上一页状态
5. 返回时不重新初始化页面
```

比如 URL 参数方式：

```text
/school/list?country=china&curriculum=ib&page=3
```

这样即使刷新页面，也能恢复筛选条件。

这也是为什么路由和页面状态会有关联。

因为有些页面状态可以直接体现在路由里。

---

# 11. 你可以这样理解它们的区别

| 概念   | 解决的问题         | 例子                                     |
| ---- | ------------- | -------------------------------------- |
| 路由   | 用户在哪个页面       | `/school/list`、`/school/detail?id=123` |
| 路由参数 | 页面需要带什么关键数据   | `id=123`、`country=china`               |
| 页面状态 | 当前页面内部是什么状态   | 筛选条件、页码、排序、滚动位置                        |
| 状态保留 | 返回页面时是否恢复之前状态 | 从详情返回列表后仍保留筛选结果                        |

---

# 12. 最简单的一句话

**路由决定你去哪一页，页面状态决定这一页当前展示成什么样。**

用户从列表页进入详情页再返回时，产品经理要决定：

> 回来后，是回到刚才那个列表状态，还是重新打开一个干净的列表页。

大多数“列表 → 详情 → 返回列表”的场景，都应该保留页面状态。
