你问的这些问题，本质是在**产品经理评估一个第三方接口/API接入方案时，需要关注的技术和业务风险点**。

其中“回调”和“异常兜底”是很多产品经理容易混淆的，我重点展开讲。

---

# 一、什么是回调？

简单理解：

> **回调 = 对方处理完成后，主动通知你结果，而不是你一直去问对方。**

也叫：

* Callback
* Webhook
* 异步通知

---

## 1. 没有回调的情况（轮询）

比如你接入一个支付接口。

流程：

```
用户付款
   ↓
你的系统调用支付平台
   ↓
支付平台处理中
   ↓
？？？
```

你的系统不知道什么时候成功。

只能不断询问：

```
你的系统：
“支付成功了吗？”

支付平台：
“没有”

过10秒：

你的系统：
“支付成功了吗？”

支付平台：
“成功了”
```

这种叫：

> 轮询（Polling）

缺点：

* 浪费接口次数
* 延迟高
* 压力大

---

## 2. 有回调的情况

流程：

```
用户付款

↓

你的系统
调用支付接口

↓

支付平台处理

↓

支付成功

↓

支付平台主动调用你的接口

↓

通知：
订单xxx支付成功

↓

你的系统更新订单状态
```

例如：

支付平台请求你的接口：

```
POST /payment/callback

{
 orderId:"10001",
 status:"SUCCESS",
 amount:100
}
```

你的系统收到：

```
订单状态：
待支付

↓

已支付
```

这就是回调。

# 二、举新能源行业例子（更贴近你的HEMS）

比如：

HaierEnergy接入第三方逆变器厂家。

场景：

设备运行数据同步。

---

## 方案A：你主动查询

你的云平台：

每分钟：

```
调用厂家接口

获取：

设备功率
电压
电流
电量
```

例如：

```
GET /device/status?id=123
```

问题：

设备10万个：

每分钟查询：

100000次

压力巨大。

---

## 方案B：厂家回调

厂家设备数据变化：

例如：

逆变器：

```
当前功率：
3500W

电池SOC：
80%
```

厂家主动调用：

你的接口：

```
POST /device/data/report

{
 deviceId:"INV001",
 power:3500,
 soc:80
}
```

你的平台：

收到数据：

↓

保存数据库

↓

APP展示

这就是设备数据回调。

# 三、什么叫异常兜底？

简单理解：

> 当正常流程失败时，系统有没有备用方案保证业务还能运行。

也叫：

容错设计。

---

## 举几个实际例子

---

# 示例1：第三方接口挂了

场景：

你的APP展示天气：

调用：

```
天气API
```

正常：

```
天气API
 ↓
返回
 ↓
APP展示
```

但是：

第三方接口挂了。

如果没有兜底：

用户看到：

```
服务器错误
加载失败
```

体验很差。

---

## 有兜底：

方案：

```
调用天气接口

↓

失败

↓

读取缓存数据

↓

展示：
"更新时间：昨天18:00"

```

用户还能看到信息。

---

# 示例2：新能源设备数据异常

你的APP：

展示：

```
家庭能源：

今日发电：
20kWh
```

数据来自设备。

突然：

设备离线。

没有兜底：

页面：

```
今日发电：
--
```

用户觉得系统坏了。

---

有兜底：

显示：

```
今日发电：
20kWh

数据更新时间：
10:30

设备当前：
离线
```

告诉用户：

> 数据不是没有，是暂时没有更新。

---

# 示例3：接口超时

比如：

调用第三方地图：

请求：

```
获取安装地址
```

正常：

2秒返回。

异常：

30秒没返回。

如果一直等：

页面卡死。

---

兜底：

设置超时时间：

```
3秒

↓

失败

↓

显示默认位置

或者

提示：
地图加载失败，请稍后重试
```

---

# 四、接口失败是否重试？

这个问题问的是：

> 如果调用失败，系统有没有自动再次尝试？

---

例如：

调用短信接口：

第一次：

```
发送失败
```

自动：

```
等待3秒

再次发送

```

再失败：

```
等待10秒

第三次发送
```

这叫：

重试机制。

---

但是不是所有接口都适合重试。

例如：

## 查询接口

可以：

```
查询设备状态
失败

↓

重新查询
```

因为不会产生副作用。

---

## 创建订单接口

谨慎。

例如：

第一次：

```
创建订单
```

服务器已经成功。

但是返回超时。

你的系统认为失败：

重新创建。

结果：

```
订单1
订单2
```

重复订单。

所以需要：

幂等设计。

---

# 五、调用是否有限频？

意思：

> 第三方接口允许你调用多少次。

例如：

厂家API：

限制：

```
1000次/分钟
```

超过：

返回：

```
429 Too Many Requests
```

---

产品经理需要知道：

* 单用户限制？
* 单设备限制？
* 企业限制？
* 每秒多少次？
* 每天多少次？

否则上线后容易：

接口被封。

---

# 六、数据是否实时？

问：

> 数据多久更新一次？

例如设备数据：

可能：

实时：

```
1秒一次
```

准实时：

```
10秒一次
```

分钟级：

```
5分钟一次
```

小时级：

```
1小时一次
```

---

新能源场景：

功率：

通常：

秒级/分钟级。

历史统计：

小时级。

---

# 七、是否有签名校验？

作用：

证明：

> 请求确实来自合法系统，没有被伪造。

例如：

攻击者：

调用你的接口：

```
POST /device/report

{
 power:999999
}
```

怎么办？

---

签名：

双方约定密钥：

```
secret=abc123
```

厂家发送：

```
deviceId=001
power=3000
timestamp=xxx

计算：

sign=MD5(参数+secret)
```

你的系统：

收到：

重新计算。

一致：

接受。

不一致：

拒绝。

---

# 八、费用如何计算？

第三方接口经常收费。

需要问：

例如：

地图：

```
免费：
10万次/月

超过：

0.5元/千次
```

短信：

```
0.05元/条
```

AI接口：

```
按token收费
```

云接口：

```
按设备数量收费
```

---

# 九、产品经理接第三方接口应该问什么？

可以形成一个接口评估表：

| 问题       | 目的       |
| -------- | -------- |
| 接口地址是什么  | 确认能力     |
| 调用频率限制多少 | 评估容量     |
| 失败是否重试   | 稳定性      |
| 有没有回调    | 确定数据同步方式 |
| 数据实时性    | 确定产品体验   |
| 是否收费     | 成本评估     |
| 是否签名校验   | 安全       |
| 接口异常怎么办  | 风险控制     |
| 有没有测试环境  | 开发验证     |
| 版本怎么升级   | 长期维护     |

---

对于你做 **HaierEnergy/HEMS、新能源设备接入**，其中最关键的几个通常是：

1. **数据同步方式：主动拉取还是厂家回调**
2. **数据实时性：秒级、分钟级还是小时级**
3. **设备离线如何处理**
4. **接口失败如何重试**
5. **数据异常如何兜底**
6. **接口限流如何扩容**

这些直接影响后续 APP 展示、设备状态判断、电站运行逻辑。
