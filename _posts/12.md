懒加载和虚拟列表什么区别？


**懒加载**和**虚拟列表**都能优化页面性能，但它们解决的问题不一样。

一句话区分：

> **懒加载：先不加载没用到的内容。**
> **虚拟列表：数据可能已经有了，但页面只渲染当前看得见的部分。**

---

# 1. 懒加载是什么？

懒加载的重点是：

> **内容先不请求、不下载、不加载，等用户快要看到或用到时再加载。**

比如学校列表页有很多学校图片。

用户刚打开页面时，只看到前 5 个学校卡片，那么下面第 50 个、第 100 个学校的图片没必要马上加载。

所以可以这样：

```text
用户打开页面
↓
只加载首屏学校图片
↓
用户往下滑
↓
快滑到下面的学校卡片时
↓
再加载对应图片
```

这就是懒加载。

---

## 懒加载主要解决什么？

解决的是：

```text
首屏加载慢
图片太多
资源太多
模块太多
接口太多
用户还没看到的内容提前加载，造成浪费
```

常见懒加载对象：

```text
图片
视频
地图
图表
弹窗内容
页面下方模块
详情页二级内容
```

---

## 举例

比如能源 APP 首页有：

```text
实时功率
今日发电量
设备状态
收益图表
历史曲线
推荐内容
告警记录
```

首屏可以先加载：

```text
实时功率
今日发电量
设备状态
```

用户往下滑到图表区域时，再加载：

```text
收益图表
历史曲线
推荐内容
```

这就是懒加载。

---

# 2. 虚拟列表是什么？

虚拟列表的重点是：

> **列表数据很多，但页面上只真正渲染用户当前看得见的那几十条。**

假设有 10000 条消息。

普通列表可能会把 10000 条都变成页面元素。

虚拟列表不会。

它只渲染屏幕里能看到的部分，比如：

```text
当前屏幕可见 20 条
上方预留 5 条
下方预留 5 条
```

也就是说，页面看起来有 10000 条，但浏览器实际只画了 30 条左右。

用户往下滚动时，前端会不断替换当前显示的内容。

---

## 虚拟列表主要解决什么？

解决的是：

```text
前端渲染太多
DOM 节点太多
滚动卡顿
浏览器内存占用高
页面点击不流畅
长列表性能差
```

常见使用场景：

```text
聊天记录
日志列表
通知列表
大规模设备列表
长评论列表
超长数据表格
```

---

# 3. 它们最大的区别

| 对比项      | 懒加载             | 虚拟列表            |
| -------- | --------------- | --------------- |
| 核心思想     | 没看到/没用到的内容先不加载  | 已有很多数据，但只渲染可视区域 |
| 重点优化     | 加载时机            | 渲染数量            |
| 主要解决     | 首屏慢、资源多、图片多     | 滚动卡、页面元素太多      |
| 是否一定用于列表 | 不一定，图片、图表、模块都可以 | 基本用于长列表         |
| 数据是否已经拿到 | 不一定，可能还没请求      | 可能已经拿到，也可能边滚边加载 |
| 典型场景     | 图片、图表、地图、下方模块   | 聊天记录、日志、大表格、长列表 |

# 4. 用学校列表举例

假设你的学校库有 5000 所学校，每所学校都有封面图。

## 懒加载解决的是图片加载问题

用户刚进入学校列表，只看到前 10 所学校。

那么后面第 100 所学校的图片不用马上加载。

```text
只加载当前屏幕附近的图片
用户滑到哪里，再加载哪里的图片
```

这叫懒加载。

它解决的是：

```text
不要一次加载 5000 张图片
```

---

## 虚拟列表解决的是页面渲染问题

如果前端一次性把 5000 所学校卡片全部渲染出来，即使图片不加载，页面也可能卡。

因为每个学校卡片都有：

```text
学校名称
国家
城市
课程标签
学费
收藏按钮
封面图区域
详情按钮
```

5000 个卡片会产生非常多页面元素。

虚拟列表会让页面只渲染当前屏幕附近的几十个卡片。

它解决的是：

```text
不要一次渲染 5000 个学校卡片
```

# 5. 用设备列表举例

假设设备总量 90000 台。

## 只用懒加载够吗？

不够。

因为懒加载通常解决图片、模块、资源加载问题。

但设备列表的问题是：

```text
数据量太多
表格行太多
前端渲染太多
查询也可能慢
```

所以设备列表首先要做：

```text
服务端分页
```

比如每页只返回 20 / 50 条。

---

## 虚拟列表适合设备列表吗？

看交互形式。

如果是后台表格，一般优先用：

```text
服务端分页
```

如果产品要求类似“无限滚动”，比如一直往下滑看设备，那么可以考虑：

```text
服务端分页 + 虚拟列表
```

也就是：

```text
后端分批返回数据
前端只渲染当前可视区域
```

---

# 6. 懒加载和虚拟列表可以一起用吗？

可以，而且经常一起用。

比如学校列表：

```text
虚拟列表：只渲染当前可见的学校卡片
懒加载：只加载当前可见学校卡片里的图片
```

也就是说：

```text
虚拟列表控制“画多少卡片”
懒加载控制“什么时候加载图片/模块”
```

两者不是互斥关系。

---

# 7. 和分页的区别也要分清

这里再补一个你容易遇到的概念：分页。

| 技术   | 解决什么             |
| ---- | ---------------- |
| 分页   | 后端每次只查一部分数据      |
| 虚拟列表 | 前端每次只渲染可视区域      |
| 懒加载  | 用户没看到/没用到的资源先不加载 |

比如 90000 条设备数据：

```text
分页：后端不要一次返回 90000 条，只返回当前页 50 条。
虚拟列表：前端不要一次渲染很多行，只渲染屏幕可见行。
懒加载：图片、图表、详情模块，用户需要时再加载。
```

最重要的一点：

> **虚拟列表不能替代服务端分页。**

如果后端一次返回 90000 条，哪怕前端用了虚拟列表，网络传输和数据库查询压力还是很大。

---

# 8. 产品经理怎么判断用哪个？

你可以按这个判断：

```text
首屏打开慢，下面内容太多 → 懒加载
图片很多，加载慢 → 图片懒加载 + 图片压缩
列表行数很多，滚动卡 → 虚拟列表
数据总量很大，接口慢 → 服务端分页
后台表格数据很多 → 优先服务端分页
聊天记录/日志流很长 → 虚拟列表
首页模块很多 → 接口分批加载 + 模块懒加载
```

# 9. PRD 里可以怎么写？

## 懒加载写法

```text
学校列表页图片采用懒加载策略，仅加载当前可视区域及其附近的学校封面图；用户滚动到对应卡片附近时再加载图片。
```

## 虚拟列表写法

```text
当消息列表数据量较大时，前端采用虚拟列表渲染，仅渲染当前可视区域内的消息项，避免一次性渲染大量 DOM 节点导致页面卡顿。
```

## 分页 + 虚拟列表写法

```text
设备列表采用服务端分页，每次加载 50 条数据；若采用连续滚动交互，前端需结合虚拟列表，仅渲染当前可视区域内的数据行。
```

# 10. 最简单总结

```text
懒加载：东西先不加载，看到再加载。
虚拟列表：东西很多，但页面只画看得见的那部分。
分页：数据很多，后端一批一批返回。
```

如果用一个现实类比：

```text
分页：仓库一次只给你拿 50 件货。
懒加载：货架下面的商品，顾客走近了再摆出来。
虚拟列表：仓库里有很多货，但店面里只展示当前货架上的几十件。
```

所以你作为产品经理要记住：

> **懒加载优化加载时机，虚拟列表优化渲染数量，分页优化数据获取规模。**
