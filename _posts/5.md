分页
虚拟列表
懒加载
骨架屏
图片压缩
接口分批加载

解释一下，分别是什么？

你问的是这几个：

```text
分页
虚拟列表
懒加载
骨架屏
图片压缩
接口分批加载
```

它们都属于**页面性能和用户体验优化手段**，但解决的问题不一样。

你可以先记住一句话：

> **分页、虚拟列表、懒加载、接口分批加载，是减少“一次性加载太多数据”；骨架屏是缓解用户等待焦虑；图片压缩是减少资源体积。**

---

# 1. 分页

## 是什么？

分页就是把大量数据拆成一页一页加载。

比如设备总共有 90000 条，不是一次全部返回，而是：

```text
第 1 页：20 条
第 2 页：20 条
第 3 页：20 条
```

接口通常这样传：

```text
page=1&pageSize=20
```

后端返回：

```text
当前页数据 list
总数量 total
```

---

## 解决什么问题？

解决的是：

```text
一次查太多数据
一次传太多数据
前端一次渲染太多数据
用户无法快速浏览
```

比如后台设备列表、订单列表、学校列表，都应该分页。

---

## 产品经理怎么理解？

分页是**列表类页面的基础设计**。

只要数据量可能很多，就应该考虑分页。

例如：

```text
设备列表
订单列表
用户列表
学校列表
内容列表
日志列表
```

---

## PRD 可以怎么写？

```text
设备列表采用服务端分页，默认每页 20 条，支持切换 20 / 50 / 100 条。
用户点击搜索后，页码重置为第 1 页。
接口需返回 total、page、pageSize、list。
```

---

# 2. 虚拟列表

## 是什么？

虚拟列表是前端优化方式。

它的意思是：

> 页面上虽然有很多数据，但浏览器只渲染用户当前看得见的那一小部分。

比如一共有 10000 条数据，但用户屏幕上只能看到 20 条。

普通列表可能会把 10000 条都渲染到页面里。

虚拟列表只渲染：

```text
当前屏幕看到的 20 条
上下预加载少量数据
```

用户往下滚动时，再动态替换显示内容。

---

## 解决什么问题？

解决的是：

```text
前端渲染卡顿
页面滚动卡顿
DOM节点太多
浏览器内存占用高
```

它不一定减少接口返回数据，主要是减少前端页面渲染压力。

---

## 和分页的区别

| 对比   | 分页        | 虚拟列表          |
| ---- | --------- | ------------- |
| 主要解决 | 数据查询和传输太多 | 前端渲染太多        |
| 发生位置 | 后端 + 前端   | 前端            |
| 用户体验 | 一页一页翻     | 连续滚动          |
| 常见场景 | 后台表格、订单列表 | 长列表、聊天记录、日志列表 |

---

## 举例

后台设备列表通常用分页。

但是聊天记录、消息列表、日志流，可能更适合虚拟列表。

因为用户希望一直往下滚，而不是点“下一页”。

---

## PRD 可以怎么写？

```text
消息列表采用虚拟滚动加载，页面仅渲染当前可视区域内的数据，避免消息过多导致页面卡顿。
```

---

# 3. 懒加载

## 是什么？

懒加载就是：

> 用户还没看到、还没用到的内容，先不加载；等需要时再加载。

比如一个页面下面有很多图片。

用户刚打开页面时，只加载首屏图片。

下面的图片等用户滚动到附近时再加载。

---

## 解决什么问题？

解决的是：

```text
首屏加载慢
一次加载资源太多
浪费流量
浪费请求
图片和模块太多导致页面慢
```

---

## 常见懒加载对象

```text
图片懒加载
列表懒加载
模块懒加载
弹窗内容懒加载
地图懒加载
图表懒加载
```

例如：页面下方有一个地图组件，用户没滚动到地图区域，就不用提前加载地图 SDK。

---

## 产品经理怎么理解？

懒加载的核心是：

> 先保证用户马上看到的内容快，再逐步加载后面的内容。

比如首页：

```text
首屏：用户信息、核心数据、关键入口
后续：推荐内容、广告位、图表、历史数据
```

不应该所有东西一起加载。

---

## PRD 可以怎么写？

```text
首页优先加载首屏核心数据，包括用户信息、今日发电量、设备状态。
推荐内容、图表模块、历史数据在用户滚动到对应区域时再加载。
```

---

# 4. 骨架屏

## 是什么？

骨架屏是一种加载占位效果。

当真实内容还没加载出来时，先展示一个灰色的页面结构。

比如：

```text
头像位置显示灰色圆形
标题位置显示灰色横条
卡片位置显示灰色块
列表位置显示灰色占位
```

它不是让接口更快，而是让用户感觉页面正在加载，并且知道大概会出现什么内容。

---

## 解决什么问题？

解决的是：

```text
用户等待焦虑
页面空白时间太长
用户不知道系统是否卡住
加载体验差
```

---

## 骨架屏和 loading 的区别

| 对比   | 普通 loading | 骨架屏      |
| ---- | ---------- | -------- |
| 展示方式 | 一个转圈       | 页面结构占位   |
| 用户感受 | 不知道要加载什么   | 知道页面大概结构 |
| 适合场景 | 短时间加载      | 页面内容加载   |
| 体验   | 一般         | 更好       |

---

## 举例

学校详情页加载时，先展示：

```text
学校封面图占位
学校名称占位
标签占位
学费信息占位
介绍段落占位
```

等接口返回后，再替换成真实内容。

---

## PRD 可以怎么写？

```text
学校详情页首次加载时展示骨架屏，占位内容包括封面图、学校名称、基础信息卡片和详情内容区域。接口返回后替换为真实数据。
```

# 5. 图片压缩

## 是什么？

图片压缩就是把图片体积变小，减少加载时间。

比如一张原图 5MB，压缩后变成 300KB。

用户肉眼看起来差别不大，但加载速度会明显提升。

---

## 解决什么问题？

解决的是：

```text
图片太大
页面加载慢
流量消耗高
移动端加载慢
首屏打开慢
```

---

## 常见做法

```text
限制上传图片大小
上传后自动压缩
生成缩略图
使用 WebP 格式
不同场景使用不同尺寸图片
图片 CDN 加速
```

比如学校列表页只需要小图，不应该加载 2000px 的高清大图。

应该使用缩略图：

```text
列表页：小图
详情页：中图
预览大图：高清图
```

---

## 产品经理怎么理解？

图片不是越清晰越好，而是要根据场景控制尺寸。

例如：

```text
头像：100x100 足够
列表封面：300x200 足够
详情页 Banner：750x400
原图：仅预览或下载时使用
```

---

## PRD 可以怎么写？

```text
用户上传学校图片后，系统自动生成缩略图和详情图。
列表页使用缩略图，详情页使用中等尺寸图片，点击预览时加载原图。
单张图片上传大小限制为 5MB，支持 JPG、PNG、WebP。
```

# 6. 接口分批加载

## 是什么？

接口分批加载就是：

> 页面需要很多数据时，不要一个接口一次返回所有内容，而是拆成多个接口，按优先级加载。

比如首页需要：

```text
用户信息
设备状态
今日发电量
收益统计
告警信息
推荐内容
消息通知
图表数据
```

如果全部放在一个接口里，任何一个慢，整个页面都慢。

分批加载可以这样做：

```text
第一批：用户信息 + 核心数据
第二批：图表数据
第三批：推荐内容
第四批：消息通知
```

---

## 解决什么问题？

解决的是：

```text
一个大接口太慢
页面必须等所有数据返回
部分数据失败影响整个页面
首屏加载慢
接口耦合太重
```

---

## 产品经理怎么理解？

接口分批加载的核心是：

> 先加载最重要的数据，让页面先可用；不重要的数据后加载。

例如能源 APP 首页：

第一优先级：

```text
电站在线状态
当前功率
今日发电量
设备告警
```

第二优先级：

```text
收益统计
历史曲线
推荐活动
资讯内容
```

第三优先级：

```text
广告
运营推荐
非核心提示
```

---

## 和懒加载的区别

| 对比   | 懒加载      | 接口分批加载         |
| ---- | -------- | -------------- |
| 核心意思 | 用到再加载    | 按优先级分批请求       |
| 关注点  | 页面资源/模块  | 接口数据           |
| 常见对象 | 图片、模块、组件 | 首页数据、图表数据、推荐数据 |
| 目的   | 减少首屏压力   | 让核心内容先出来       |

它们经常一起使用。

---

## PRD 可以怎么写？

```text
首页数据采用分批加载策略：
第一批加载电站状态、实时功率、今日发电量；
第二批加载收益统计和历史曲线；
第三批加载推荐内容和运营信息。
若第二、三批接口失败，不影响首页核心数据展示。
```

---

# 7. 这几个概念放在一起怎么理解？

你可以这样记：

| 技术手段   | 主要解决什么     | 产品经理关注点                 |
| ------ | ---------- | ----------------------- |
| 分页     | 数据太多，一次查不完 | 每页多少条、是否显示总数、搜索后是否回到第一页 |
| 虚拟列表   | 前端渲染太多导致卡顿 | 是否需要连续滚动、大数据量列表是否流畅     |
| 懒加载    | 没看到的内容先不加载 | 首屏先展示什么，后面内容什么时候加载      |
| 骨架屏    | 等待时体验差     | 加载时用户看到什么               |
| 图片压缩   | 图片太大导致慢    | 上传限制、缩略图、不同场景用不同尺寸      |
| 接口分批加载 | 一个页面依赖太多数据 | 哪些数据优先加载，哪些失败不影响主流程     |

# 8. 用一个真实页面举例

假设你做一个“学校库列表页”。

页面内容包括：

```text
学校列表
学校图片
筛选条件
分页
推荐学校
广告位
收藏状态
```

可以这样设计：

## 分页

学校列表每页显示 20 条。

```text
避免一次返回几千所学校。
```

## 图片压缩

列表页只用学校缩略图。

```text
避免加载高清大图。
```

## 懒加载

用户滚动到某张学校卡片附近时，再加载图片。

```text
避免首屏加载所有图片。
```

## 骨架屏

列表接口还没返回时，先显示学校卡片骨架。

```text
避免页面空白。
```

## 接口分批加载

第一批加载学校列表。

第二批加载推荐学校和广告位。

```text
避免推荐模块影响学校列表展示。
```

## 虚拟列表

如果是无限滚动且列表特别长，可以用虚拟列表。

```text
避免前端一次渲染太多学校卡片。
```

# 9. 再用设备列表页举例

假设你的设备列表有 90000 条。

比较合理的设计是：

```text
服务端分页：每次只查 20 / 50 条
筛选条件接口：安装商、状态、地区单独加载
列表接口：只返回当前页必要字段
详情接口：点击设备后再查详情
图片/图标：使用轻量资源
加载时：表格骨架屏或 loading
```

如果做成一次返回全部 90000 条，即使用虚拟列表，后端查询和网络传输也会很慢。

所以要注意：

> 虚拟列表不能替代服务端分页。

这是一个很重要的点。

# 10. 最容易混淆的几个点

## 分页和虚拟列表不是一回事

分页是控制“查多少、传多少”。

虚拟列表是控制“页面渲染多少”。

---

## 懒加载不是骨架屏

懒加载是“晚点加载”。

骨架屏是“加载期间显示占位”。

---

## 图片压缩不是懒加载

图片压缩是让图片变小。

懒加载是让图片晚点加载。

两者可以一起用。

---

## 接口分批加载不是分页

分页是同一种数据分多页。

接口分批加载是不同模块的数据分批请求。

比如：

```text
分页：学校列表第1页、第2页、第3页
接口分批：学校列表、推荐学校、广告位、收藏状态分开加载
```

---

# 11. 产品经理怎么判断该用哪个？

你可以这样判断：

```text
数据条数很多 → 分页
页面渲染卡 → 虚拟列表
首屏东西太多 → 懒加载 / 接口分批加载
等待体验差 → 骨架屏
图片加载慢 → 图片压缩
一个接口太大太慢 → 接口分批加载
列表连续滚动很长 → 虚拟列表
```

---

# 12. 一句话总结

```text
分页：把大量数据拆成一页一页查。
虚拟列表：只渲染屏幕上看得见的数据。
懒加载：没看到、没用到的内容先不加载。
骨架屏：数据没回来前先展示页面结构占位。
图片压缩：把图片变小，减少加载时间。
接口分批加载：把一个页面的数据按优先级分批请求。
```

作为产品经理，你不需要知道它们具体代码怎么写，但你要知道：

> **它们分别解决哪一层的问题：数据量、渲染量、资源体积、加载时机、等待体验、接口拆分。**
